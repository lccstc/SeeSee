# 报价解析器重构计划：从通用正则到群组模板

## 问题背景

当前解析器使用通用正则匹配所有客人群的报价，导致大量解析错误：
- catch-all 行（"The rest of countries"）被误收为报价
- 数量-面额混合行（"10-495 【5X】"）面额识别错误
- 计算行（"CAD 500 * 4= ₦388800"）被当成报价
- 正则越堆越多，边界情况无法穷举

**根本原因：试图用一个通用规则解析所有不同格式的报价。**

## 新方案：群组模板匹配（"机场广播"模式）

### 核心理念

每个客人群的报价格式是**固定的结构**，变动的只有变量值。

类比机场广播：
```
固定结构: "{航班号} 从 {出发地} 飞往 {目的地}，预计 {时间} 到达，票价 {价格}"
变动变量: 航班号、出发地、目的地、时间、价格
```

报价单同理：
```
客人A 固定结构: "{卡种} / {国家} {面额} 【{倍数}X】 ¥ {价格} ₦ {奈拉}"]
变动变量: 国家、面额、倍数、价格、奈拉
```

### 优势
- 不需要通用正则——每个模板只处理一种格式
- 解析准确率极高——结构已知，只采数字
- 维护成本低——格式变了，改模板；价格变了，自动采集
- 不会泛化出错——一个模板的错误不影响其他群

## 只采集客人组

**Set 5/6/7/8 = 客人报价组**，只有这些群的消息需要解析上墙。
其他群的消息不进解析器，直接忽略。

## 模板定义设计

### 模板格式（草案）

每个模板是一个简单的描述文件，定义：
1. **行模式** — 每一行报价长什么样
2. **变量槽位** — 哪些位置是变量，变量是什么类型
3. **映射规则** — 变量对应到数据库的哪个字段

```yaml
# 客人A - Apple 横白卡模板
group: "客人A群名称"
card_type: "Apple"           # 固定值
form_factor: "横白"          # 固定值

lines:
  # 模式: "100/150 【50X】 ¥ 5.31 ₦ 1075.81"
  - pattern: "{denomination} 【{multiplier}X】 ¥ {price} ₦ {naira}"
    variables:
      denomination: {type: "amount_range", map_to: "amount_range"}
      multiplier:   {type: "integer", map_to: "multiplier"}
      price:        {type: "float", map_to: "price", primary: true}
    country_from_header: true  # 国家从上方 section header 获取

  # 模式: section header "——————Apple USA ——————"
  - pattern: "——————{card_type} {country} ——————"
    variables:
      country: {type: "country", map_to: "country_or_currency"}
    is_section_header: true
```

```yaml
# 客人B - 简单格式模板
group: "客人B群名称"

lines:
  # 模式: "Apple&iTunes/CAD 500 ¥ 3.84"
  - pattern: "{card_type}/{country} {denomination} ¥ {price}"
    variables:
      card_type:    {type: "card_type", map_to: "card_type"}
      country:      {type: "country", map_to: "country_or_currency"}
      denomination: {type: "integer", map_to: "amount_range"}
      price:        {type: "float", map_to: "price", primary: true}
```

### 变量类型

| 类型 | 说明 | 示例 |
|------|------|------|
| `float` | 浮点数 | 5.31, 3.84 |
| `integer` | 整数 | 100, 500 |
| `amount_range` | 面额范围 | "100/150", "10-495" |
| `country` | 国家/币种 | "USD", "CAD", "UK" |
| `card_type` | 卡种 | "Apple", "Steam" |
| `multiplier` | 倍数 | 50, 100 |
| `text` | 自由文本 | 限制说明 |
| `optional_*` | 以上任意类型的可选版本 | 可以缺失 |

### 模板存储

使用现有 `quote_group_profiles.template_config` 字段（JSON 格式），无需新建表。

```json
{
  "version": 1,
  "type": "fixed_template",
  "card_type": "Apple",
  "form_factor": "横白",
  "sections": [
    {
      "header_pattern": "———{card_type} {country}———",
      "header_variables": {"country": "country"},
      "line_pattern": "{denomination}【{multiplier}X】¥{price}₦{naira}",
      "line_variables": {
        "denomination": "amount_range",
        "multiplier": "integer",
        "price": "float",
        "naira": "float"
      }
    }
  ],
  "skip_patterns": [
    "The rest of the countries",
    "Cooperation & suggestions",
    "Recommend friends"
  ]
}
```

## 实施步骤

### 阶段 1：只采客人组（立即可做）
- 在解析入口加过滤：只处理 Set 5/6/7/8 的群组消息
- 其他群消息不进解析器，直接跳过
- 效果：立刻消除 90% 的噪音报价

### 阶段 2：模板引擎
- 设计模板格式（与 Leo 确认每个客人的报价格式）
- 实现模板匹配引擎（字符串匹配 + 变量提取，不用正则）
- 实现 skip_patterns 过滤（catch-all、footer 等）

### 阶段 3：逐群录入模板
- 对每个客人组，人工定义一个模板
- 先做 2-3 个客人的模板验证引擎
- 再批量推广到所有客人组

### 阶段 4：迁移旧解析器
- 模板引擎稳定后，逐步替换旧的通用解析器
- 保留旧解析器作为 fallback（非客人组的消息）
- 最终完全移除通用正则逻辑

## 需要 Leo 确认的问题

1. **客人组的 WhatsApp chat_id 列表** — Set 5/6/7/8 具体是哪些群？
2. **每个客人的报价格式** — 每种格式给一个真实样例，我来写模板
3. **模板格式是否需要更复杂** — 比如一个群里有多种报价格式（如 Apple 和 Steam 在同一个群）
4. **异常处理** — 模板匹配失败的消息怎么处理？进异常区还是直接丢弃？

## 验收标准

- [ ] 阶段 1：非客人组的消息不再产生报价
- [ ] 阶段 2：模板引擎能正确解析 3 个以上客人的报价
- [ ] 阶段 3：所有客人组都有对应模板
- [ ] 阶段 4：旧解析器代码移除，报价墙上无错误数据
