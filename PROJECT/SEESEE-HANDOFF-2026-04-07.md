## 3. 项目一句话定义

SeeSee 不是普通聊天机器人，也不是单纯记账页面。

它本质上是一个：

**礼品卡中介业务中的，聊天入口驱动的交易、记账、回溯、排错与风控辅助系统。**

---

## 4. 真实业务场景

Leo 当前的核心业务是礼品卡中介与报价撮合。

真实链路大致是：

1. 从客人群收集报价
2. 汇总成价格表
3. 向供应商群广播收卡价
4. 从供应商收卡
5. 转发给客人
6. 客人加载完成后，分别给供应商和客人加账
7. 最后做对账、追差额、定位错误

利润来自中间差价。

所以这个系统的目标，不是做一个会聊天的机器人，而是：

- 减少人工差错
- 提高回溯能力
- 提高排错速度
- 最终减少业务链路中的错账、漏账、错群、错价

---

## 5. 当前最真实的痛点

当前最痛的不是“流量不够”，也不是“没有页面”，而是忙的时候容易出错，而且出错后很难快速定位。

典型问题包括：

1. **价格加错**
   - 本来应该按某个报价成交，结果记账价格错了

2. **账单加错群**
   - 本该记在某个客户群 / 供应商群，结果记到错误群里

3. **漏加单**
   - 业务发生了，但某一边没有入账

4. **客户加了，供应商没加**
   - 或反过来

5. **重复加单 / 多加单**
   - 同一笔业务被重复记录

6. **双边都记错，但各自公式成立**
   - 例如每一边 amount × rate = rmb 都算得平，但整体业务仍然是错的

Leo 真正需要系统帮忙抓出来的，是上面这些业务错误。

## 7. 必须明确：当前系统还没有真正解决业务痛点

这是这次交接里最重要的一点。

当前做出来的东西不是“完全没用”，但它们主要解决的是：

- 留证据
- 查链路
- 做单笔公式检查

它们还没有解决 Leo 真正关心的业务错误。

### 当前系统能做什么

- 知道一条消息最终记成了哪笔交易
- 知道一条交易是不是被改过
- 检查一条交易自身的 amount / rate / rmb 是否算得平
- 在 workbench 上看到某个卡种的差额变大了

### 当前系统做不到什么

- 自动判断客户和供应商是否正确配对
- 自动识别“客户加了但供应商没加”
- 自动识别“漏加单”
- 自动识别“账单加错群”
- 自动识别“重复加单 / 多加单”
- 自动识别“价格偏离当时报价”
- 自动识别“双方都记错但各自内部仍然算得平”

### 当前异常体系的本质

当前异常更多是“单笔技术性检查”，而不是“业务级异常归因”。

也就是说，它现在更像：

**A 面台账结构化 + 技术预警初级版**

还不是：

**真正的业务对账与异常归因系统**

当前这条线如果继续补，很容易继续产出：

换句话说：

### trace / reconciliation 线更偏什么
- 事后回看
- 事后排查
- 单笔留痕
- 单笔自检

### Leo 当前真正更需要什么
- 事前 / 事中基准
- 业务规则判断
- 配对逻辑
- 错价识别
- 错群识别
- 漏加识别

所以继续深挖 trace 面板的优先级应该下降。

它可以保留，但应被视为：

**二级取证工具，而不是当前主线。**

## 9. 为什么“报价系统”其实是核心

当前越来越明确的一点是：

**很多真实业务错误，不引入报价基准，根本抓不出来。**

### 例子 1：价格加错
如果系统不知道这个群、这个卡种、这个时点本来应该按什么价格成交，那就无法判断实际记账价格是否偏离。

### 例子 2：双边都记错但公式成立
客户侧和供应商侧都可能把 amount / rate / rmb 记成了彼此自洽的数字。

这种情况下：
- 单笔公式检查过关
- workbench 差额未必明显异常
- 但业务上仍然错了

如果有报价系统，系统才能知道：

- 当时报价是多少
- 合理成交区间是什么
- 实际价格是否偏离
- 偏离发生在哪一边

### 例子 3：错群 / 错单
如果报价中心记录了某个群、某个人、某个卡种在某个时点的有效报价与上下文，它就能为“这笔账到底应属于哪条链路”提供参照。

所以报价系统的意义，不只是做一个页面看报价，而是要为后续：

- 错价识别
- 配对判断
- 漏加定位
- 业务归因

提供基准层。

- 业务角色
- 卡种

- 时间

1. 一笔交易如何回挂到报价基准
2. 如何判断成交价偏离当时报价

### F. 报价系统要帮助抓哪些异常
报价系统未来必须至少能支持识别这些异常：

- 错价
- 错群

- 重复加单
- 双边都错但表面公式成立

## 12. 当前建议的工作顺序

### 第一阶段：只做方案，不急着大写代码
先产出一份真正靠谱的方案文档，至少明确：

1. 报价系统最小闭环是什么
2. 数据模型怎么设计
3. 解析规则怎么设计
4. 页面最小结构是什么
5. 与 workbench / reconciliation 怎么衔接
6. 分阶段怎么做，哪些先不做

### 第二阶段：先做最小闭环，不做大而全
建议最小闭环是：

1. 能识别报价消息
2. 能提取结构化报价字段
3. 有“待审核报价队列”
4. 有“当前有效报价表”
5. 交易能挂回报价基准
6. 至少能对“错价”提供第一版判断

### 第三阶段：再考虑跟对账中心融合
等报价基准真正稳定后，再反过来增强：

- workbench 异常归因
- reconciliation 的业务级异常过滤
- 自动配对 / 半自动配对
