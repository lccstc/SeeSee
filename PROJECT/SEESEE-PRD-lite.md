# SeeSee 生产化轻量 PRD

## 1. 目标
把 SeeSee 从“能记账”推进成“能回溯、能排错、能解释”的聊天驱动交易与总账系统。

一句话目标：
> 让每一笔账，都能从总账一路追溯回原始聊天消息，并逐步减少人工差错和排查成本。

## 2. 当前真实痛点
目前最痛的不是流量，而是忙的时候容易出错，而且出错后很难快速回溯。
典型问题：
- 价格记错
- 账单记错
- 群主发错
- 漏加单
- 差额定位慢

## 3. 当前阶段的推进顺序

### 阶段 A，证据链
目标：先把原始消息完整录下来，并且能查到。
产出：
- 原始消息入库
- 原始消息查询能力

### 阶段 B，结构化解析
目标：让系统能区分普通聊天、命令、报价、交易型消息。
产出：
- 消息分类
- 结构化字段提取
- 解析结果可回看

### 阶段 C，关联与回溯
目标：把“原始消息 -> 解析结果 -> 交易记录”串起来。
产出：
- 原始消息与交易记录的关联查询
- 出错时可回放整条链路

### 阶段 D，差额追踪
目标：当账不对时，能快速定位是哪一环出了问题。
产出：
- 差额来源定位
- 可视化回溯页面或视图

## 4. 当前阶段明确不做
这些现在不是第一优先级：
- 不先做花哨页面
- 不先做复杂自动聊天
- 不先做大规模流程放大
- 不先让本地模型直接做复杂业务裁决

## 5. 当前技术协作原则
- GPT-5.4 负责拆任务、验收、收口
- 本地 31B coding worker 负责轻量、边界清楚、可测试的编码任务
- 一个任务一个新 job，做完并验收后再开下一轮
- 每次改动前，先说明“下一步做什么、会带来什么变化”
- 先做可验证的小步成果，不开盲盒

## 6. 当前第一优先级
当前优先级仍然是：
1. 补证据链
2. 做结构化解析
3. 做关联回溯
4. 再做差额追踪
