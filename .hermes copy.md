## 1. 项目目标

- SeeSee 是一个礼品卡中介报价撮合与总账系统，不是单纯的网页项目。
- 当前真实运行架构是：WeChat 监听器 + WhatsApp 监听器 + Core 核心层。
- 两个监听器只负责接消息、归一化消息、把消息送进 Core，再执行 Core 返回的动作。
- 当前真实业务主线是：平台消息 -> Core -> 报价墙 / 记账 / 对账 / 动作回流。
- 当前阶段没有完整的大蓝图，不要假设项目已经设计完毕。实际工作方式是：做一点、验证一点、再迭代一点。
- 当前最高优先级的产品文档是：`/Users/newlcc/SeeSee/repo/PROJECT/报价墙prd.md`
- 当前阶段的目标不是“做一个万能 AI 解析器”，而是先把报价墙这条链路做稳、做可追溯、做可人工接管。
- 长期目标是让机器人逐步参与真实业务，包括：按报价墙自动报价、接收卡片、转发给客人、确认加载成功、加账单、与供应商结账等机械化操作。
- 但是请明确区分：
  - 当前已实现：双监听适配层、统一 Core runtime、总账中心、报价墙 MVP、异常区、模板、字典、短回复上下文
  - 未来方向：供应商机器人自动报价与全流程自动化

## 2. 当前阶段

- 当前项目不是“学习 demo”，而是真实运营系统，涉及钱、报价、消息、机器人和数据库。
- 当前最重要的子系统是报价墙 P1/P1.5：稳定采集、稳定解析、稳定展示、稳定追溯。
- 当前方向是“群固定模板 + 报价字典 + 异常区人工补洞”，不是“通用正则无限扩张”，也不是“AI 直接进主链路”。
- 当前事实源是客人群原始消息，不是同事整理后的报价单。
- 当前 WeChat 和 WhatsApp 都已经拆成薄监听/执行层，业务命令和业务状态全部走 Core。
- Core 是项目里最重要的核心层。只要涉及业务判断、命令解析、报价采集、记账、动作返回，优先看 Core，不要先钻监听器。
- 当前 Hermes 回答和修改代码时，必须优先理解“现在已经上线了什么”以及“现在真正想修的痛点”，不要按未来终局反推一整套大重构。

## 3. 业务概念解释

- Core：统一业务核心层。正式入口是 `POST /api/core/messages`，它负责接收所有平台归一化消息、处理命令、记账、报价采集、生成动作。
- 监听器 / 适配器：WeChat 和 WhatsApp 两边的薄接入层。它们不应该承载主业务逻辑。
- 客人群：报价来源群。报价墙当前主要采集这类群。
- 供应商群：后续会逐步接入自动报价和自动执行，但当前不是主链路。
- 报价墙：按标准结构聚合后的“当前最高价行情墙”，不是原始消息流水。
- 排名：同一 SKU 下所有来源群的报价排序，不等于主墙。
- 历史切片：旧价格不会丢，用于之后做错价检测和回查。
- 异常区：模板不匹配、缺上下文、未知字典项、限制行、异常价格等都会先到这里。
- 短回复上下文：先建一个询价上下文，再把同群的短回复价格补成完整报价。
- 群模板：按 `platform + chat_id` 绑定的解析规则。当前这是主路线。
- 报价字典：国家/币种、卡种、形态等别名映射。支持内置映射、自定义映射、群级覆盖。
- 内部标准值：系统内部统一口径，例如 `Code` 最终可能归一到 `代码`。Hermes 必须把“用户输入值”和“内部标准值”区分清楚，不要混淆。

## 6. 当前架构的最重要事实

- 这是“双监听 + Core 核心驱动”架构，不是前后端分离 Web app。
- Core 是唯一业务核心。WeChat 和 WhatsApp 都不是各自一套业务系统。
- 正式消息入口是 `POST /api/core/messages`。
- `UnifiedBookkeepingRuntime` 会先记录 incoming message，再跑报价捕获，再跑记账服务。
- WeChat 当前是 remote core mode：监听消息 -> `send_envelope()` -> Core -> 执行动作 -> 轮询 Core outbound actions。
- WhatsApp 当前也是薄适配层：监听消息 -> normalize -> `POST /api/core/messages` -> 执行动作，并轮询/确认 Core outbound actions。
- 所有业务命令都应视为“Core 命令”，不是 WeChat 命令或 WhatsApp 命令，平台差异只体现在监听和发送层。
- 报价捕获主流程在 `QuoteCaptureService.capture_from_message(...)`。
- 报价数据会落入：
  - `quote_documents`
  - `quote_price_rows`
  - `quote_parse_exceptions`
  - `quote_inquiry_contexts`
  - `quote_group_profiles`
  - `quote_dictionary_aliases`

- `/quotes` 页面不是一个简单列表，它承载了：
  - 主墙
  - 客人群模板
  - 短回复上下文
  - 异常区
  - 排名弹窗

- `/quote-dictionary` 页面承载字典维护：
  - 内置映射
  - 自定义映射
  - 群级覆盖
  - 内部标准值
  - 管理口令保护

## 7. 报价系统当前原则

- 当前不追求万能解析。
- 主路线是：群模板优先，字典辅助，异常兜底。
- 模板不匹配时，优先进入异常区，不要静默丢失，也不要硬猜入墙。
- 主墙展示的是“归一化后同条件下的当前最高价”，不是所有历史行。
- 低价、重复、被更高价支配的行，应退到排名或历史，不应污染主墙。
- 历史切片必须保留，因为后续要做错价检测与回查。
- 短回复必须依赖上下文，不要靠猜。
- 字典和模板都是运营工具，不只是技术配置。

## 8. 当前实现上的关键约束

- 当前正式 runtime 只接受 PostgreSQL DSN；不要默认 SQLite 是正式事实源。
- 当前两个监听器都依赖 Core 可用；如果 Core 没启动、Token 不一致、endpoint 不通，两个平台都会表现异常。
- 当前很多报价采集逻辑只针对客人组，且当前实现里会按 group number 过滤客人群。
- 当前报价捕获还带有“只有配了模板配置的群才解析”的保守门槛。调试时必须先确认群模板是否已配。
- 当前时间、超时、展示时间有现实影响。任何时间处理改动都要非常谨慎。
- 任何会影响“错误价格上墙”的改动，都属于高风险。

## 9. 当前产品优先级

- 第一优先级：报价墙链路稳定
  - 真消息进来能采到
  - 模板群能正确解析
  - 主墙整洁
  - 异常不丢
  - 历史可追溯

- 第二优先级：运营补洞工具顺手
  - 异常区能一键建模板
  - 字典页能看清已有映射、输入值、内部标准值
  - 能快速补国家、卡种、形态别名

- 第三优先级：为后续自动化保留干净结构
  - 供应商报价机器人
  - 自动收卡、转发、确认、加账、结算

## 10. 当前不是重点的事情

- 不要现在为了“最终自动化”去提前做庞大抽象层。
- 不要把 AI 当主解析链路。
- 不要默认每条异常都要人工审核后才能入墙。
- 不要把同事整理后的报价单当当前权威事实源。
- 不要把未来供应商机器人能力写成当前系统已具备能力。

## 12. 代码约定

- 这是 Python 项目，使用项目内 `.venv`。
- 当前测试主入口是 `unittest`，不要默认是 `pytest`。
- 新代码优先沿用现有模式，不要凭空创建新框架。
- Web 页面当前是服务端拼 HTML + 内联 JS，不要轻易假设它是 React/Vue。
- 业务逻辑优先放在 `bookkeeping_core/`，不要把业务规则塞进页面层。
- 路由层尽量留在 `bookkeeping_web/app.py`，页面展示留在 `pages.py`。
- 报价规则优先模板化、字典化，不要把每个群的特殊情况都硬写成散落的正则补丁。

## 13. 高风险区域

- PostgreSQL schema 和真实账务数据
- Core API / outbound actions / 机器人发消息链路
- 报价解析、字典归一化、主墙聚合
- 时间处理、时区处理、超时判断
- 会删除或覆盖历史报价的操作
