# 2026-03-20 财务控制系统设计稿

## 0. 文档状态

- 状态：`Draft / 已完成设计确认`
- 范围：一期设计，不含实现代码
- 当前目标：先替代手工抄表、手工对齐、手工回查
- 当前明确不做：价格中心、AI 自动报价、完整财务利润模型、自动改账

## 1. 设计结论

一期采用`首页驾驶舱 + 下钻详情`方案，而不是做一个把所有信息塞满的大页面。

系统由 4 个主要页面组成：

1. `首页驾驶舱`
2. `跑账详情工作台`
3. `客户 / 供应商总表`
4. `跑账历史页`

设计原则：

1. 首页只保留控制信号和入口，不承担全部细节。
2. 详细页按 `PC 高信息密度` 设计，默认就是表格、筛选、历史列表和回查区。
3. 全系统围绕`跑账区间`组织财务口径。
4. 同时保留`时间区间分析`能力，用于月度复盘、卡种分析和趋势对比。
5. 一期就纳入`未跑账期间异常预警`，尽快发现错账，而不是等跑账后才发现。

## 2. 视觉与交互方向

视觉方向采用：

- `彭博终端 + 苹果官网`

落地原则：

1. 首页气质克制、高级、强信号。
2. 详细页保留专业后台的高密度信息组织。
3. 不是营销页，不做大面积空白展示页。
4. 不是传统后台，不做堆满表格却没有优先级的工作台。

具体分工：

- 首页：少而准
- 详细页：密而清

## 3. 核心业务前提

### 3.1 角色映射

一期归属映射按当前已确认规则执行：

- `1 = 内部群`
- `2 / 3 / 4 = 供应商群`
- `5 / 6 / 7 / 8 = 客户群`

归属优先级：

1. `主体/群手工指定`
2. `/set 数字映射`
3. `未归属`

`未归属群`不进入正式财务汇总，只进入待处理和异常入口。

### 3.2 跑账方式

一期按`全局批量跑账`建模。

也就是说，一次跑账不是单个群自己的动作，而是一个全局财务动作，对应一个明确的时间范围。

### 3.3 当前余额

`客户 / 供应商总表`里的`实际剩余金额`采用：

- `实时当前余额`

不是“最近一次跑账后的快照余额”。

## 4. 数据硬定义

### 4.1 关键概念

一期底层围绕 4 类核心对象设计：

1. `群`
2. `客户/供应商主体`
3. `跑账区间`
4. `卡种统计行`

### 4.2 群

定义：

- 实际聊天入口与采集单位

字段至少包括：

- `platform`
- `chat_id`
- `chat_name`
- `group_num`
- `business_role`
- `status`
- `capture_enabled`

### 4.3 客户 / 供应商主体

定义：

- 一个业务主体，可以挂多个群

一期规则：

1. 一期先按群运行
2. 一期支持手工把多个群归并到同一个主体
3. 不做自动归并

### 4.4 跑账区间

定义：

- 一次全局扎账对应的时间范围

展示格式：

- `2026-01-01 09:30 → 2026-01-02 13:30`

建议字段：

- `id`
- `start_at`
- `end_at`
- `closed_at`
- `closed_by`
- `note`
- `has_adjustment`
- `mx_snapshot_version`

边界规则建议：

- 采用半开区间：`start_at < record_time <= end_at`

这样相邻两个跑账区间不会重复计数。

### 4.5 卡种统计行

定义：

- 在某个范围内，按`卡种 + 刀数`聚合出的一行统计

说明：

- 如果原始文本里带 `单张面额 * 数量`，则额外保存
- 但主业务口径仍然围绕`卡种 + 刀数 + 汇率`

建议字段：

- `scope_type`
- `scope_id`
- `business_role`
- `card_type`
- `usd_amount`
- `rate`
- `rmb_amount`
- `unit_face_value`
- `unit_count`
- `raw_text`

## 5. 正式记录模型

### 5.1 业务记录主字段

每条业务记录至少需要：

- `角色`：客户 / 供应商
- `卡种`
- `刀数`
- `汇率`
- `人民币金额`
- `记录时间`
- `来源群`
- `所属主体`
- `原始文本`
- `解析版本`

可选增强字段：

- `单张面额`
- `数量`

### 5.2 刀数、数量、面额定义

这是一期最关键的术语修正。

- `刀数`
  不是数量
  是`总美金额`

- `数量`
  不是主字段
  只在原文明确出现时才保存

- `单张面额`
  只在原文明确出现时才保存

例子：

- `steam50*10`
  应结构化为：
  - `card_type = steam`
  - `unit_face_value = 50`
  - `unit_count = 10`
  - `usd_amount = 500`

- `+500st5.0`
  应结构化为：
  - `card_type = steam`
  - `usd_amount = 500`
  - `rate = 5.0`

### 5.3 汇率是一等字段

一期必须把`汇率`作为一等字段。

原因：

1. 你们的利润本质来自汇差。
2. 如果页面只显示人民币金额，不显示汇率，就看不出利润来源。
3. 错账排查也离不开汇率。

## 6. 指标硬定义

### 6.1 主体实时余额

公式：

- `主体实时余额 = 该主体下所有有效群的实时余额合计`

时间口径：

- 实时

展示页面：

- `客户 / 供应商总表`

### 6.2 当前总余额

公式：

- `当前总余额 = 所有已归属客户主体实时余额合计 + 所有已归属供应商主体实时余额合计`

排除项：

- `内部群`
- `未归属群`

展示页面：

- `首页驾驶舱`

### 6.3 客户汇率金额

公式：

- `客户汇率金额 = Σ 客户记录.usd_amount × 客户记录.rate`

时间口径：

- 跑账区间或分析区间

展示页面：

- `跑账详情工作台`
- `跑账历史页`

### 6.4 供应商汇率金额

公式：

- `供应商汇率金额 = Σ 供应商记录.usd_amount × 供应商记录.rate`

时间口径：

- 跑账区间或分析区间

展示页面：

- `跑账详情工作台`
- `跑账历史页`

### 6.5 跑账利润

公式：

- `跑账利润 = 客户汇率金额 - 供应商汇率金额`

等价写法：

- `Σ usd_amount × (客户汇率 - 供应商汇率)`

说明：

- 一期页面统一使用`跑账利润`这个词
- 不使用宽泛的“利润”
- 它不是完整财务利润，不包含付款、返利、坏账等更复杂维度

展示页面：

- `首页驾驶舱`
- `跑账详情工作台`
- `跑账历史页`

### 6.6 今日已实现利润

公式：

- `今日已实现利润 = 今天完成的所有跑账区间.跑账利润之和`

说明：

- 不是按今天自然流水乱算
- 如果今天发生了多次跑账，全部计入

展示页面：

- `首页驾驶舱`

下钻路径：

- 点击进入`跑账详情工作台`

### 6.7 总刀数

公式：

- `总刀数 = Σ 当前所选范围内所有记录.usd_amount`

说明：

- 这里的“刀数”是总美金额
- 不是张数

展示页面：

- `首页驾驶舱`
- `跑账详情工作台`
- `跑账历史页`

### 6.8 高风险负余额金额

规则：

- 当前先按 `客户主体余额 < -1000`

公式：

- `高风险负余额金额 = Σ abs(客户主体余额)`，条件是该主体余额 `< -1000`

展示页面：

- `首页驾驶舱`

### 6.9 本月主体业务金额

定义：

- 客户主体：本月客户汇率金额
- 供应商主体：本月供应商汇率金额

默认时间范围：

- 当月

展示页面：

- `客户 / 供应商总表`

### 6.10 本月卡种总刀数

公式：

- `本月卡种总刀数 = Σ 当前区间内该卡种记录.usd_amount`

用途：

- 回答“这个月 XX 卡做了多少刀”

展示页面：

- `跑账历史页` 的区间分析区

## 7. 页面结构

### 7.1 首页驾驶舱

定位：

- 只发出信号，不承担全部工作

固定展示：

- `当前总余额`
- `今日已实现利润`
- `高风险负余额金额`
- `今日收款`

次级信号：

- `总刀数`
- `未归属群数`
- `待修正账期数`
- `待配对 / 疑似错账数`

入口：

- `客户 / 供应商总表`
- `跑账历史`

点击路径：

1. 点击 `今日已实现利润` -> `跑账详情工作台`
2. 点击 `客户 / 供应商总表` -> 主体总表
3. 点击 `跑账历史` -> 历史复盘页
4. 点击异常卡片 -> 对应筛选结果

### 7.2 跑账详情工作台

定位：

- PC 高密度页面
- 同时承担 `当前工作期监控` 与 `历史跑账回查`

左侧区：

- 历史跑账区间列表

第一项固定为：

- `当前未跑账工作区`

其余项为：

- 已完成的历史跑账区间

顶部汇总：

- `客户总刀数`
- `客户汇率金额`
- `供应商总刀数`
- `供应商汇率金额`
- `跑账利润`
- `待配对记录数`
- `疑似错账数`

主区域：

1. `卡种明细表`
2. `可疑记录表`
3. `原始记录 / 修正记录 / 操作日志`
4. `异常与修正`

默认逻辑：

1. 从首页利润卡进入时，默认打开今天最近完成的跑账区间
2. 直接进入该页时，默认打开`当前未跑账工作区`

### 7.3 客户 / 供应商总表

定位：

- 主体搜索、排序、余额查看、分组查看页面

默认视图：

- 默认时间范围 = `当月`
- 默认排序 = `本月业务金额` 从高到低

主表字段：

- `主体名`
- `角色`
- `分组`
- `关联群数量`
- `实时当前余额`
- `本月业务金额`
- `本月总刀数`
- `最近活动时间`
- `风险状态`

支持操作：

- 搜索 `主体名 / 群名 / 分组 / 角色`
- 排序 `本月业务金额 / 本月总刀数 / 实时当前余额 / 最近活动时间`
- 手工归并多个群到同一主体

### 7.4 跑账历史页

定位：

- 宏观复盘与时间区间分析

默认视图：

- 默认时间范围 = `当月`

顶部汇总：

- `总刀数`
- `客户汇率金额`
- `供应商汇率金额`
- `跑账利润`

主区分成两块：

1. `跑账区间列表`
2. `月度 / 区间卡种分析`

区间列表字段：

- `跑账区间`
- `总刀数`
- `客户汇率金额`
- `供应商汇率金额`
- `跑账利润`
- `修正次数`
- `异常次数`
- `操作人`

卡种分析能力：

- 搜索 `卡种关键词`
- 排序 `总刀数 / 客户汇率金额 / 供应商汇率金额 / 跑账利润`

## 8. 搜索与排行

### 8.1 一期不做首页全局搜索

一期不做首页全局搜索。

原因：

1. 先把页面内搜索和排序做稳
2. 避免首页再次变杂
3. 更符合 PC 高密度操作场景

### 8.2 主体搜索与排行

所在页面：

- `客户 / 供应商总表`

默认统计周期：

- `当月`

支持：

- 搜索：`主体名 / 群名 / 分组 / 角色`
- 排序：`本月业务金额 / 本月总刀数 / 实时当前余额 / 最近活动时间`

### 8.3 卡种搜索与排行

所在页面：

- `跑账历史页` 的区间分析区
- `跑账详情工作台` 的卡种明细区

默认统计周期：

- `当月`

支持：

- 搜索：`卡种关键词`
- 排序：`总刀数 / 客户汇率金额 / 供应商汇率金额 / 跑账利润`

## 9. 一期异常预警与排错

### 9.1 一期目标

一期目标不是自动改账，而是：

1. 尽快发现错账
2. 快速缩小范围
3. 给出最可疑记录
4. 支持回看原始文本和修正轨迹

### 9.2 当前未跑账工作区

这是`跑账详情工作台`的默认第一项。

作用：

- 监控当前工作期间的录入情况
- 不等跑账后才发现问题

顶部信号建议：

- `待配对记录数`
- `疑似错账数`
- `缺汇率 / 解析失败数`
- `客户总刀数`
- `供应商总刀数`
- `当前预估跑账利润`

### 9.3 匹配与预警规则

主匹配维度：

- `卡种 + 刀数`

增强校验维度：

- `单张面额`
- `数量`

状态定义：

- `待配对`
  只录了一边，另一边尚未出现

- `疑似错账`
  两边都出现了，但 `卡种` 或 `刀数` 不一致

- `待处理`
  汇率缺失、解析失败、归属不明

说明：

- 金额不作为主匹配条件
- 因为客户金额和供应商金额本来就会因汇率不同而不同

### 9.4 当前工作区主表字段

- `状态`
- `记录时间`
- `角色`
- `主体`
- `来源群`
- `卡种`
- `刀数`
- `汇率`
- `人民币金额`
- `配对对象`
- `差异说明`

### 9.5 一期承诺做到的能力

1. 立刻发现不平
2. 告诉你差在哪个卡种、哪个刀数
3. 给出最可疑记录对
4. 支持点击回看原始记录、修正记录、操作日志

### 9.6 一期不承诺的能力

1. 100% 自动断言一定是哪一条错
2. 自动替用户改账

## 10. 采集策略

### 10.1 主事实源

- `结构化交易记录` 是主事实源

### 10.2 快照事实

- 每次跑账时保存一份 `/mx` 结果，作为该跑账区间的卡种快照

### 10.3 双轨并存

一期允许：

1. `结构化交易记录`
2. `跑账快照`

并存。

使用原则：

1. 页面统计优先用结构化事实
2. 结构化事实暂时不完整时，用跑账快照补齐卡种统计
3. 所有统计结果必须标明时间范围

## 11. 一期明确不做

一期明确不做：

1. 价格中心
2. AI 自动报价
3. 首页全局搜索
4. 自动归并客户/供应商主体
5. 自动改账
6. 完整财务利润模型
7. 基于主体级的一定正确的利润归属排名

说明：

- 主体页面的一期排序以 `业务金额 / 总刀数 / 当前余额` 为主
- 更复杂的主体利润口径放到后续阶段

## 12. 风险与后续阶段

### 12.1 一期主要风险

1. 原始输入格式不完全稳定
2. 结构化解析能力还不够统一
3. `/mx` 快照与结构化记录可能出现暂时性差异
4. 主体归并完全依赖人工，治理流程必须清晰

### 12.2 二期自然延伸方向

1. 主体利润归属与更细匹配
2. 自动建议可疑错账位置
3. 自动建议客户/供应商主体归并
4. 价格中心
5. AI 辅助报价与决策

## 13. 实现优先级建议

推荐按下面顺序落地：

1. `跑账区间模型`
2. `记录模型修正：卡种 / 刀数 / 汇率 / 人民币金额`
3. `跑账详情工作台`
4. `当前未跑账工作区异常预警`
5. `客户 / 供应商总表`
6. `跑账历史页`
7. `页面内搜索与排序`

---

本设计稿反映的是当前已确认的一期方向：

- 首页克制
- 详细页高密度
- 利润来自汇差
- 刀数是总美金额
- 一期即纳入异常预警
- 搜索与排行先做页面内能力
