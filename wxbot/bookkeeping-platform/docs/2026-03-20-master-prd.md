# 2026-03-20 Master PRD：账期驱动型财务控制系统

## 0. 文档状态

- 状态：`Draft / 已确认主线，待拆子 PRD`
- 当前阶段产出：`Master PRD`
- 当前阶段明确不做：代码实现、UI 开发、比价系统开发、AI 决策上线

## 1. 背景与业务目标

### 1.1 当前业务现实

现有业务同时运行在 `WhatsApp` 与 `WeChat` 两个聊天入口上。

当前最真实、最紧迫的业务问题，不是“智能比价”或“AI 决策”，而是：

1. 财务每天需要手动录入账单。
2. 财务需要人工核对余额、扎账结果与异常差额。
3. 财务大量时间消耗在翻群消息、抄表、对齐口径上。
4. 现有最小总账页面只能看数据，不能真正承担工作台职责。

### 1.2 当前阶段目标

当前阶段只解决一个问题：

`先把财务从手工录入和手工对齐中解放出来。`

这意味着系统第一阶段的目标是：

1. 建立稳定的财务数据底座。
2. 建立按账期组织的财务控制台。
3. 让首页能直接处理高频财务动作。
4. 让老板能在同一首页看到资金风险与当日已实现利润。

### 1.3 非当前阶段目标

以下内容属于后续阶段，不进入当前阶段实现：

- 比价系统
- 客户 / 供应商最优价识别
- 机会利润
- AI 自动识别
- AI 推荐与自动决策

### 1.4 一句话定位

这是一个`受控采集 + 账期控制 + 财务工作台`系统，而不是普通聊天机器人后台，也不是当前阶段的比价平台。

## 2. 战略边界

### 2.1 阶段顺序

当前阶段顺序已经固定：

1. `先跑通财务系统`
2. `再做比价系统`
3. `最后才考虑 AI`

### 2.2 为什么不能提前做比价和 AI

如果现在提前把比价和 AI 拉进来，会带来四个直接风险：

1. 需求边界膨胀，财务系统迟迟无法落地。
2. 财务口径未稳定，后续利润与推荐结果会漂。
3. 用户需要同时理解财务、报价、AI 三套逻辑，认知负担过高。
4. 项目容易变成“每块都做一点，但没有一块真正可用”。

### 2.3 当前阶段的成功标准

第一阶段成功，不是看“功能多不多”，而是看：

1. 财务是否明显减少手工录入。
2. 财务是否明显减少手工对齐。
3. 首页是否真的能承担日常工作台职责。
4. 老板是否能一眼看到资金风险和当天已实现利润。

## 3. 用户角色与核心场景

### 3.1 老板

核心问题：

- 当前账期资金是否安全。
- 当前有哪些异常群或差额。
- 今天已经赚了多少钱。
- 哪些事项需要立刻处理。

核心动作：

- 查看首页顶部摘要。
- 下钻到异常群与账期明细。
- 通过标签与角色快速切换业务口径。

### 3.2 财务

核心问题：

- 哪些群待扎账。
- 哪些群账期异常或差额未平。
- 当前余额与经营解释是否一致。
- 哪些数据需要人工修正。

核心动作：

- 在首页处理待扎账队列。
- 在首页确认异常并提交修正。
- 通过实时筛选快速定位群。
- 从摘要一路追到账期和原始消息。

### 3.3 业务员

第一阶段业务员不是首页主角色，但其输入会影响系统质量。

职责：

- 为群打数字标签。
- 必要时覆盖群的客户 / 供应商角色。
- 协助纠正群归属和基础资料。

## 4. 产品定位与设计原则

### 4.1 首页原则

首页不是报表页，而是控制页。

首页必须满足：

1. 先看异常、待办、风险和当天已实现利润。
2. 先支持操作，再支持浏览。
3. 任何摘要指标都能下钻到群、账期、消息。

### 4.2 时间主轴

全系统首页主轴按`账期`组织。

说明：

- 财务动作天然围绕账期展开。
- 老板仍需要看“当日已实现利润”，但这是账期控制台顶部摘要的一部分，不取代账期主轴。

### 4.3 设计原则

1. 事实优先，不做不可追溯汇总。
2. 先解决控制，再扩展分析。
3. 先做稳定规则与人工确认，再考虑自动化增强。
4. 先做实时筛选，再做复杂组合管理。
5. 一期只做已打标业务群，不碰无关聊天噪音。

## 5. 一期范围与非范围

### 5.1 一期范围

一期只做“财务采集底座 + 账期控制”闭环。

包括：

1. 已打标业务群的受控采集。
2. 记账消息与群元数据进入统一事实层。
3. 群标签、角色映射与群级覆盖管理。
4. 按账期组织的混合首页。
5. 首页直接处理扎账、异常、修正。
6. 数字勾选实时筛选。
7. 常用筛选预设保存。
8. 当天已实现利润展示。

### 5.2 一期明确不做

一期明确不做：

- 报价解析与标准化
- 比价系统
- 客户 / 供应商最优价推荐
- 机会利润主指标
- AI 自动判定作为主链路
- AI 推荐与自动决策
- 全平台所有群无差别采集
- 独立预付款台账

### 5.3 后续阶段预留

虽然当前不做，但系统设计需要为后续阶段保留边界：

- 报价消息采集
- 比价系统
- 经营分析深化
- AI 辅助识别与推荐

这些都属于路线图，不属于当前阶段交付范围。

## 6. 数据域与事实源

### 6.1 事实源原则

第一阶段采用`受控保留`原则。

这里的“受控保留”不是所有聊天全量，而是：

- 在已打标业务群内，
- 对记账相关消息、群元数据相关消息，
- 保留原文、元数据、筛选结果、结构化结果与解析版本。

不采集无关闲聊，不把所有平台群一股脑入库。

### 6.2 一期核心数据域

第一阶段只统一两类核心事实：

1. `群与标签元数据`
2. `账务事实`

### 6.3 GroupProfile

`GroupProfile` 是群的一等对象，至少覆盖：

- `group_key`
- `platform`
- `chat_id`
- `chat_name`
- `tag_number`
- `role_source`
- `business_role`
- `role_override`
- `status`
- `capture_enabled`
- `updated_at`

规则：

- `tag_number` 是群分组数字标签。
- `business_role` 默认由数字标签映射得出。
- 允许少量群做 `group-level override`。
- `capture_enabled` 用于控制该群是否进入采集范围。

### 6.4 MessageCaptureRecord

`MessageCaptureRecord` 是可追溯能力的基础。

至少覆盖：

- `message_key`
- `group_key`
- `platform`
- `chat_id`
- `chat_name`
- `sender_id`
- `sender_name`
- `sent_at`
- `raw_content`
- `message_type`
- `capture_reason`
- `filter_result`
- `parser_version`
- `normalized_payload`
- `review_status`

规则：

- `message_type` 初期至少区分：`accounting`、`group_meta`
- `capture_reason` 记录为什么这条消息被纳入事实层
- `filter_result` 记录规则筛选是否命中
- `review_status` 支持后续人工纠错与重跑解析

### 6.5 AccountingEvent

`AccountingEvent` 是统一账务事件对象。

至少覆盖：

- `event_key`
- `group_key`
- `event_type`
- `occurred_at`
- `source_message_key`
- `settlement_id`
- `amount_delta`
- `balance_after`
- `created_by`
- `review_state`

初期支持的事件类型：

- 交易新增
- 交易删除 / 撤销
- 扎账完成
- 人工修正
- 群标签变更

### 6.6 SettlementPeriod

`SettlementPeriod` 是首页主轴对象。

至少覆盖：

- `settlement_id`
- `group_key`
- `period_start_at`
- `period_end_at`
- `status`
- `opening_balance`
- `income_total`
- `expense_total`
- `closing_balance`
- `variance_amount`
- `manual_adjustment_count`
- `owner`

规则：

- 账期不是自然日主轴，而是以扎账切分。
- 当前打开的未完成账期与历史已完成账期都应能在产品上表达。

### 6.7 FilterPreset

`FilterPreset` 支持首页实时筛选与复用。

至少覆盖：

- `preset_id`
- `name`
- `tag_numbers`
- `business_roles`
- `platforms`
- `created_by`
- `updated_at`

产品规则：

- 默认交互是勾选即实时刷新。
- 常用组合允许保存为预设。
- 不再把“组合汇总”设计成先建后看的主流程。

### 6.8 未来扩展预留

未来可能扩展到：

- 报价消息采集
- 价格标准化
- 比价与机会利润
- AI 辅助识别与推荐

这些能力当前只作为路线图预留，不进入一期规格约束。

## 7. 关键口径

### 7.1 客户 / 供应商

- 默认由数字标签映射得出。
- 允许群级覆盖。
- 不再依赖平台做角色判断。

### 7.2 Prepayments

- 一期不建独立预付款台账。
- 作为余额的经营解释展示：
  - `余额 > 0`：对方在我方有预付款
  - `余额 < 0`：对方欠我方款

### 7.3 当天赚了多少钱

- 一期按`已实现利润`展示。
- 仅统计真实进入账务事实与账期事实的利润。
- 不把机会利润混入首页主指标。

## 8. 首页与导航设计

### 8.1 首页总体结构

首页采用三层结构：

1. 顶部老板摘要区
2. 中部财务工作台
3. 底部下钻区

### 8.2 顶部老板摘要区

该区域优先服务老板，但财务也能快速理解整体状态。

第一屏必须有：

- 当前账期总余额
- 当前账期异常差额
- 待处理事项数
- 风险群数
- 当天已实现利润

设计要求：

- 资金与风险优先级最高。
- 利润必须和风险同屏出现。
- 每张摘要卡都可下钻。

### 8.3 中部财务工作台

该区域是首页主操作区。

必须包含：

- 待扎账队列
- 异常队列
- 人工修正入口
- 当前账期关键动作

第一阶段首页允许的高频动作：

- 扎账
- 异常确认
- 人工修正
- 进入明细追溯

复杂操作可以进入详情页，但首页不能只看不能动。

### 8.4 底部下钻区

该区域负责从摘要下钻到业务明细。

必须支持：

- 数字标签勾选实时筛选
- 客户 / 供应商角色筛选
- 平台筛选
- 群级列表
- 账期明细列表

组合筛选产品规则：

- 默认模式为勾选即刷新。
- 可保存预设。
- 取消现有“新建一次、罗列一次”的主要交互模式。

### 8.5 导航建议

第一阶段建议采用轻导航：

- 首页：混合控制台
- 群管理：群资料、标签、角色覆盖
- 消息中心：采集记录、筛选结果、人工修正
- 账期明细：历史账期与群级下钻

比价页与 AI 页当前不进入一级导航，也不作为一期页面目标。

## 9. 风险与待确认项

### 9.1 数字标签映射表

当前已确认“角色默认由数字标签映射”，但映射表本身仍需单独明确。

后续子 PRD 需要固定：

- 哪些数字默认是客户
- 哪些数字默认是供应商
- 哪些数字属于内部或特殊角色

### 9.2 已实现利润公式

当前只确认首页展示“当天已实现利润”，但尚需在子 PRD 中固定：

- 利润是否按账期已落账计算
- 是否允许人工修正影响利润
- 当天口径与账期口径冲突时如何展示

### 9.3 记账消息筛选与归类

当前只确认一期围绕记账消息与群元数据展开，后续子 PRD 需要明确：

- 哪些消息视为记账事实
- 哪些消息视为群元数据变更
- 误判如何人工纠正
- 历史消息如何重放

### 9.4 历史数据回填

当前仓库已有部分账务历史导入能力，但更完整的群元数据历史仍未整理。

后续需要评估：

- 历史消息是否可补采
- 补采成本是否值得
- 历史缺口是否接受

## 10. 路线图

### 10.1 第二阶段：比价系统

在财务系统跑通之后，再进入比价系统建设：

- 报价消息采集
- 卡种字典
- 市场 / 国家标准化
- 面值区间标准化
- 客户价 / 供应商价对照
- 最优价识别
- 机会利润计算

### 10.2 第三阶段：AI 决策与运营优化

在比价系统跑通之后，再考虑 AI：

- 自动识别增强
- 智能比价
- 成交建议
- 异常预警
- 经营优化建议

## 11. 子 PRD 拆分顺序

Master PRD 确认后，按以下顺序拆子 PRD：

### 11.1 子 PRD 1：账期控制台

目标：

- 把首页、账期对象、异常队列、修正流程、筛选预设写透。

### 11.2 子 PRD 2：财务数据底座与群治理

目标：

- 把群标签、角色覆盖、消息采集、原文保留、历史回填写透。

### 11.3 子 PRD 3：比价系统

目标：

- 把报价采集、标准化、对照、机会利润与经营口径写透。

### 11.4 子 PRD 4：经营与 AI 决策

目标：

- 把推荐、预警与 AI 介入边界写透。

## 12. 验收标准

Master PRD 被视为达标，至少满足：

1. 系统边界已经明确，不再把本项目误解成单纯 UI 改版。
2. 一期范围清楚，未把比价系统与 AI 提前塞进当前阶段。
3. 首页对象、主轴、角色和动作已经明确。
4. 数据域已经覆盖群元数据与账务事实，并对未来扩展保留边界。
5. 子 PRD 拆分顺序清楚，可直接进入下一轮详细规格设计。

## 13. 结论

本项目当前阶段应被定义为`财务系统建设项目`，而不是比价或 AI 项目。

正确推进顺序是：

1. 先确认 Master PRD
2. 再拆财务相关子 PRD
3. 先落地财务系统
4. 财务系统稳定后再做比价系统
5. 比价系统稳定后再考虑 AI

因此，本文件是当前阶段唯一正式产出，也是后续实现与评审的上游约束。
