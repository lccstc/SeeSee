# 高中生想接盘 3

## 先用一句话说人话

这次 P2.5，我们做的事情可以理解成：

**把“记账这件事”从几条松散的小路，整理成一条更统一、更能复现、更方便测试的大路。**

以前系统里有几种不同来源的数据：

- 真正运行时收到的消息
- 兼容旧系统导入进来的同步事件
- 测试时手工伪造的 mock 数据

它们虽然最后都想进入同一本账，但走法不完全一样，所以会出现一个问题：

**同样看起来像一笔交易，最后生成出来的账期快照、卡种统计、历史排行，不一定稳定。**

P2.5 的目标，就是先把这个地基打平。

---

## 这次到底解决了什么问题

可以把系统想成一个记账工厂。

以前的情况像这样：

1. 微信上来的消息，工厂 A 处理
2. 旧兼容同步事件，工厂 B 处理
3. 测试数据，工厂 C 处理

虽然最后都想进“总账仓库”，但每个工厂写单子的格式不完全一样。

结果就是：

- 有些字段会丢
- 有些统计只能在某一条路径上出现
- `/workbench` 和 `/history` 这种页面需要的“卡片统计数据”，不一定能稳定生成
- SQLite 能过，fake PostgreSQL 不一定也能过

所以这次我们不是在做“新页面”，也不是在做“新业务规则”，而是在做：

**统一记账入口的格式 + 统一测试回放方式 + 补齐旧数据快照需要的统计。**

---

## 这次做了哪 5 件大事

## 1. 先定义了一个统一的“交易格式”

新增了一个统一的数据结构，名字叫：

- `StructuredTransactionRecord`

你可以把它理解成“记账工厂统一要求的入库单”。

以前不同路径写交易，虽然字段差不多，但不是同一个明确的格式。

现在统一之后，一笔交易至少会更清楚地带上这些信息：

- 平台
- 群
- 发消息的人
- 消息 ID
- 金额
- 分类
- 汇率
- 人民币值
- 原始文本
- 解析版本
- 美金额
- 面值
- 张数

意义很简单：

**以后不管这笔交易来自哪里，进数据库之前，先变成同一种“表格格式”。**

这一步主要在这里：

- `wxbot/bookkeeping-platform/bookkeeping_core/ingestion.py`

---

## 2. 让运行时消息和兼容同步事件都走同一条入账路

这一步可以理解成：

以前两个窗口收单，现在虽然窗口还在，但它们都必须把单子交给同一个总入口。

我们没有改掉系统对外的边界：

- `POST /api/core/messages` 还是正式运行时入口
- `POST /api/sync/events` 还是兼容旧系统的入口

但我们改了它们内部的做法：

- 运行时收到交易消息后，不再自己随手往数据库塞一条
- 兼容同步事件收到 `transaction.created` 后，也不再自己随手塞

现在它们都会先组装成统一的 `StructuredTransactionRecord`，再通过统一持久化入口写入数据库。

这件事的价值是：

- 以后更不容易出现“微信路径有这个字段，兼容路径没这个字段”
- 测试更容易验证两条路径到底是不是一致
- 后面如果要继续升级 parser 或回放逻辑，改动点会更集中

---

## 3. 做了一套可复用的 mock replay 回放工具

这一块是 P2.5 很关键的成果。

因为 `/workbench` 和 `/history` 这种页面，不是只看“有没有交易”就行，它们要看的是：

- 账期
- 群快照
- 卡种统计
- 历史排行

以前测试这些东西时，有不少地方是“手工拼数据”：

- 这边塞一条交易
- 那边手动改几个字段
- 再去手搓账期

这样做的问题是：

- 重复代码多
- 不同测试造的数据口径不一致
- 以后别人接手时很难看出“什么才是标准测试场景”

所以这次我们抽了一套共享工具，主要在：

- `wxbot/bookkeeping-platform/tests/support/bookkeeping_replay.py`

它做的事情可以理解成：

1. 先帮你准备一个标准的测试场景
2. 再模拟 runtime 回放消息
3. 再补上这个场景需要的分析字段
4. 最后关账，生成账期快照和卡种统计

这样一来：

- `test_analytics.py`
- `test_webapp.py`
- `test_ingestion_alignment.py`

都能用同一套“回放工具”来造数据。

这就像以前每个同学自己抄一份实验步骤，现在老师终于发了标准实验讲义。

---

## 4. 补齐了旧数据回填时缺失的卡种统计

这是给“老数据”补课。

系统里有一个逻辑叫：

- `backfill_legacy_periods()`

你可以把它理解成：

“以前已经结算过的旧账，现在补生成账期快照。”

以前这个逻辑能补出：

- `accounting_periods`
- `period_group_snapshots`

但有时补不出：

- `period_card_stats`

这会导致一个很现实的问题：

表面上看账期已经有了，但像 `/workbench`、`/history` 这种页面需要的卡种统计还是空的。

所以这次我们补上了这件事，而且还处理了一个升级场景：

- 不只是“第一次回填”能生成 card stats
- 连“以前已经回填过旧快照，但当时没写 card stats”的情况，现在再次执行也能补回来

这个修复很重要，因为它解决的不是“新数据”，而是“历史脏数据/半成品数据”。

---

## 5. 确认 fake PostgreSQL 也能跑通，并补了说明文档

这一步是“别只在自己电脑上看着像能用”。

这次我们不只验证了 SQLite，也验证了 fake PostgreSQL 路径。

意思不是去连真实 PostgreSQL 数据库，而是：

- 用现有的 fake `psycopg` 夹具
- 模拟 PostgreSQL 风格的数据访问
- 确认同样的 mock replay 路径能生成：
  - workbench 的 `card_stats`
  - history 的 `card_rankings`
  - 并且 JSON 序列化没问题

同时 README 也补了 P2.5 的说明，明确讲清楚：

- `/api/core/messages` 是 live runtime 边界
- `/api/sync/events` 只是兼容路径
- settlement bootstrap 只是兼容验证，不是 P2.5 主验收路径
- P2.5 的 mock replay 验证命令怎么跑

这很像做完实验后，终于把“这套实验到底怎么复现”写进实验报告，而不是只留在脑子里。

---

## 现在系统比以前更稳定在哪里

如果你把这次改动理解透，核心提升就 4 点：

### 1. 入口格式统一了

不管消息来自 runtime、兼容同步、还是测试回放，至少“进账本之前”更像同一种东西。

### 2. 分析页面更容易稳定复现

`/workbench` 和 `/history` 需要的 card stats / rankings，现在有了更统一的测试回放方式。

### 3. 旧数据不再只补半套

以前 legacy backfill 可能只有 period snapshot，没有 card stats。
现在这条缺口也补上了。

### 4. SQLite 和 fake PostgreSQL 都验证过

这意味着我们不是只在一种数据库风格下碰巧通过。

---

## 这次没有做什么

这个也很重要，防止你误会。

P2.5 **没有**做这些事：

- 没有把 WeChat / WhatsApp 的真实生产接入方式重做
- 没有做真正的生产数据库迁移
- 没有发明更复杂的新交易语法
- 没有做 P3、P4 那些更大的治理/异常流程
- 没有重做页面视觉

也就是说：

**这次主要是“修路、铺地基、补测试、补兼容”，不是“盖新楼”。**

## 如果你是高中生接盘，建议先这样理解代码

不要一上来就试图看完整个项目。

建议按这个顺序理解：

### 第一步：先看统一交易格式

看：

- `wxbot/bookkeeping-platform/bookkeeping_core/ingestion.py`

你要先搞懂：

- 一笔交易现在长什么样
- runtime builder 和 sync builder 分别怎么把数据变成统一格式

### 第二步：再看运行时和兼容路径怎么接进来

看：

- `wxbot/bookkeeping-platform/bookkeeping_core/service.py`
- `wxbot/bookkeeping-platform/bookkeeping_core/sync_events.py`

你要搞懂：

- runtime 消息从哪里进
- sync 事件从哪里进
- 它们最后怎么走到同一个持久化入口

### 第三步：再看账期怎么生成

看：

- `wxbot/bookkeeping-platform/bookkeeping_core/periods.py`

你要搞懂：

- 关账时怎么生成 period snapshot
- 怎么生成 card stats
- legacy backfill 怎么补历史账期

### 第四步：最后看测试回放工具

看：

- `wxbot/bookkeeping-platform/tests/support/bookkeeping_replay.py`

你要搞懂：

- 测试场景是怎么搭的
- 为什么 workbench/history 的测试现在可以复用这套 helper

---

## 如果你以后继续接着做，最可能的下一步是什么

按现在的状态，后面最自然的方向可能有 3 个：

### 方向 1：把测试 helper 和真实 runtime 的分析字段能力继续拉近

现在 mock replay helper 里还有“集中回填分析字段”的做法。
这对测试很实用，但它说明：

**真实 runtime 原生生成这些分析字段的能力，还没有完全成为唯一来源。**

以后如果继续演进，一个方向就是让真实路径更原生地产出这些字段，减少测试补丁感。

### 方向 2：继续推进 P3 / P4

等 P2.5 地基稳了，后面才能更安心地做：

- 主体治理
- 角色映射
- 异常工作流

### 方向 3：继续补更强的跨数据库回归

现在 fake PostgreSQL 已经能跑通，但如果以后数据库逻辑更复杂，还需要继续盯：

- 类型兼容
- 时间字段兼容
- JSON 序列化兼容

---

## 最后给接盘人的一句话

如果你现在觉得“代码很多、名词很多、看不懂”，这是正常的。

你只要先记住这次 P2.5 的核心目的就够了：

**把记账入口格式统一起来，把页面依赖的快照和卡统计稳定产出来，把测试回放做成能重复使用的标准工具。**

你不是要一次性理解全项目。

你只需要先回答这 3 个问题：

1. 一笔交易现在统一长什么样？
2. 它怎么从不同入口进入数据库？
3. 它最后怎么变成 `/workbench` 和 `/history` 能看的数据？

这 3 个问题搞懂了，这次 P2.5 你就算接住了。
