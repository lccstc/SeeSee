### Locked Decisions
[VERIFIED: .planning/phases/02.1-real-exception-corpus-candidate-coverage/02.1-CONTEXT.md] Copied verbatim below.

- **D-01:** This phase is not pure research. It must build the real exception corpus and also directly raise candidate hit rate on the highest-volume groups during the same phase.
- **D-02:** Coverage priority is `top 8 exception-heavy chats first`, because they currently represent the majority of the open pool. At the same time, the full open exception pool should receive a lighter-weight index so long-tail shapes are not invisible.
- **D-03:** Live exception pool evidence is the main planning dataset. Planning and implementation should optimize against real collected customer messages, not synthetic examples or prompt-crafted demos.
- **D-04:** The corpus is refreshable. If the current material library is not sufficient during implementation, the phase should pull more samples directly from the live exception site/API instead of treating the first harvested slice as complete.
- **D-05:** Gold-standard samples must be message-level and row-level. For each curated sample, the system should capture: valid quote lines, normalized row fields, which rows should be `publishable`, `held`, or `rejected`, and whether the message behaves more like `full_snapshot`, `delta_update`, or remains unresolved.
- **D-06:** A sample is not “gold” if it only marks quote-like lines. It must also express the intended standard fields and the row/message outcomes needed by downstream validator and publisher phases.
- **D-07:** The gold set should be usable for replay, fixtures, and regression tests rather than existing only as prose notes.
- **D-08:** For high-frequency groups, the user remains the final authority on gold-standard answers.
- **D-09:** For lower-frequency groups and long-tail samples, the agent may propose candidate answers first, but those answers still require confirmation before they become trusted gold material.
- **D-10:** The project should prefer “agent proposes, human confirms” over waiting for perfect manual labeling of the entire pool before implementation starts.
- **D-11:** This phase may use deterministic templates, deterministic parsing rules, section splitters, and line classifiers.
- **D-12:** Models may be used to explore local candidate strategies, but models may not define business rules, publish rules, validator rules, or final fact outcomes.
- **D-13:** Any model-assisted improvement must still resolve into explicit system logic, persisted evidence, replayable behavior, and regression coverage.
- **D-14:** “Done” means both sides move forward together: the material-library loop becomes real, and the top exception-heavy groups show visible candidate-generation improvement against live samples.
- **D-15:** It is not enough to produce a pretty corpus document. The phase must also ship replayable sample sets, fixtures, and concrete code-level candidate improvements.
- **D-16:** It is also not enough to chase hit rate without durable materials. Improvements must be grounded in a reusable corpus and regression surface.
- **D-17:** New or low-volume groups do not need full coverage in this phase, but they should not stay completely opaque. The phase should add a conservative fallback path that extracts what can be justified and leaves the rest in the exception pool.
- **D-18:** Long-tail fallback may help candidate generation, but if scope is weak or context is missing, the correct outcome remains “do not publish”.
- **D-19:** This phase must not weaken the current “宁可不上墙，不可误上墙” posture just to make long-tail groups look more covered.
- **D-20:** This phase does not change validator custody boundaries from Phase 2.
- **D-21:** This phase does not centralize publish authority or remove fact-write bypasses; that remains Phase 3 work.
- **D-22:** This phase does not let templates, agents, or prompts directly publish active quote facts.

## Summary

Phase 02.1 should be planned as a candidate-layer hardening phase with two deliverables that move together: a refreshable real exception corpus and targeted parser/template improvements against the most common live shapes [VERIFIED: .planning/phases/02.1-real-exception-corpus-candidate-coverage/02.1-CONTEXT.md]. The live pool is concentrated enough to support this strategy: the research baseline captured 226 open exceptions across 25 chats, and the top 8 chats contribute 159 of them, or roughly 70% [VERIFIED: .planning/research/REAL-EXCEPTION-CORPUS.md]. A fresh spot-check of the live endpoint still returns open `missing_group_template` and `strict_match_failed` records with the same kinds of multi-section and update-style boards, so the corpus cannot be treated as frozen input [VERIFIED: live exception API].

The codebase already has the correct custody boundaries for this phase. Runtime and replay both persist `QuoteCandidateMessage`/`QuoteCandidateRow` bundles into `quote_documents` and `quote_candidate_rows`, then compute validator-owned `publishable_rows` through `quote_validation_runs` and `quote_validation_row_results` [VERIFIED: bookkeeping_core/quote_candidates.py, bookkeeping_core/database.py, bookkeeping_core/quote_validation.py]. Existing web flows for exception harvest/result save also replay into candidate-only documents and assert `mutated_active_facts=False`, which means Phase 02.1 can safely increase candidate coverage without taking on publisher work early [VERIFIED: bookkeeping_web/app.py, tests/test_webapp.py].

The planning implication is that corpus packaging and parser work should share one regression surface. Gold material should not be stored as prose-only notes; it should be encoded as replayable message fixtures with expected candidate rows, expected validator outcomes, and unresolved vs. candidate snapshot labels for later phases [VERIFIED: .planning/phases/02.1-real-exception-corpus-candidate-coverage/02.1-CONTEXT.md, bookkeeping_web/app.py]. Candidate improvements should focus on section splitting, quote-vs-rule-vs-inquiry classification, header-scope carry-forward for naked numeric rows, template bootstrap for missing-template chats, and conservative deterministic fallback for long-tail groups [VERIFIED: .planning/research/REAL-EXCEPTION-CORPUS.md, bookkeeping_core/template_engine.py, bookkeeping_core/quotes.py].

**Primary recommendation:** Plan Phase 02.1 around one shared loop: harvest live exceptions -> curate gold fixtures -> harden deterministic candidate parsing for top-8 chats -> replay against validator-owned outcomes -> package regressions in repo tests and fixtures [VERIFIED: .planning/phases/02.1-real-exception-corpus-candidate-coverage/02.1-CONTEXT.md, .planning/research/REAL-EXCEPTION-CORPUS.md, bookkeeping_core/database.py, bookkeeping_web/app.py].

**When to use:** Use this as the default candidate-side splitter/classifier before any long-tail fallback inference, especially for mixed boards with manual rules and inquiry text [VERIFIED: .planning/research/REAL-EXCEPTION-CORPUS.md, tests/test_template_engine.py].

### Anti-Patterns to Avoid

- **Corpus as prose notes only:** A document-only corpus cannot drive replay or regression, and it violates the locked decision that gold material must be fixture-usable [VERIFIED: .planning/phases/02.1-real-exception-corpus-candidate-coverage/02.1-CONTEXT.md].
- **Parser hints treated as publish authority:** Phase 2 intentionally decoupled parser `row_publishable` hints from validator-owned `final_decision`, so planning must not regress that boundary [VERIFIED: .planning/STATE.md, bookkeeping_core/quote_validation.py, tests/test_quote_validation.py].
- **New parser stack before exploiting current template engine:** Existing `group-parser-v1`, `strict-section-v1`, and result-preview flows already cover section derivation and replay, so replacing them would add risk without solving the identified bottleneck first [VERIFIED: bookkeeping_core/template_engine.py, bookkeeping_web/app.py].
- **Long-tail “coverage” by guessing context:** Context-poor lines are supposed to stay in exceptions or be held/rejected, not promoted into publishable candidates [VERIFIED: .planning/phases/02.1-real-exception-corpus-candidate-coverage/02.1-CONTEXT.md, .planning/PROJECT.md].

## Common Pitfalls

### Pitfall 1: Treating `missing_group_template` and `strict_match_failed` as the same failure
**What goes wrong:** Planning collapses template bootstrap work and section/classifier work into one generic “make parser smarter” bucket [VERIFIED: .planning/research/REAL-EXCEPTION-CORPUS.md].  
**Why it happens:** Both appear in the same exception pool, but they represent different remediation lanes: no template vs. template present but brittle [VERIFIED: .planning/research/REAL-EXCEPTION-CORPUS.md, bookkeeping_core/quotes.py].  
**How to avoid:** Plan separate tasks for template bootstrap and for template-survival improvements such as section splitting, rule filtering, and header carry-forward [VERIFIED: .planning/phases/02.1-real-exception-corpus-candidate-coverage/02.1-CONTEXT.md].  
**Warning signs:** A plan item says “improve parser generally” without naming whether it targets missing-template groups or strict-match groups [VERIFIED: .planning/research/REAL-EXCEPTION-CORPUS.md].

### Pitfall 2: Building gold samples that stop at quote-like line marking
**What goes wrong:** The resulting corpus cannot measure validator-facing outcomes or support later publisher phases [VERIFIED: .planning/phases/02.1-real-exception-corpus-candidate-coverage/02.1-CONTEXT.md].  
**Why it happens:** It is faster to label lines than to encode normalized rows plus publishable/held/rejected outcomes [VERIFIED: .planning/phases/02.1-real-exception-corpus-candidate-coverage/02.1-CONTEXT.md].  
**How to avoid:** Require every gold sample to include message metadata, normalized row fields, expected candidate rows, and expected validator outcomes [VERIFIED: .planning/phases/02.1-real-exception-corpus-candidate-coverage/02.1-CONTEXT.md, bookkeeping_core/quote_candidates.py, bookkeeping_core/quote_validation.py].  
**Warning signs:** A corpus row has only `source_line` and no normalized SKU fields or final row outcome [VERIFIED: bookkeeping_core/quote_candidates.py, bookkeeping_core/quote_validation.py].

### Pitfall 3: Letting replay packaging drift away from the real runtime path
**What goes wrong:** Fixes look good in fixture scripts but fail in runtime/replay because they bypass the actual document/validation tables [VERIFIED: bookkeeping_web/app.py].  
**Why it happens:** Offline scripts are easier to write than running through the same `record_quote_candidate_bundle()` and `record_quote_validation_run()` path as the app [VERIFIED: bookkeeping_core/database.py].  
**How to avoid:** Make every regression fixture replay through the same candidate and validator services used by runtime and web replay [VERIFIED: bookkeeping_web/app.py, tests/test_webapp.py].  
**Warning signs:** A proposed regression harness never mentions `quote_documents`, `quote_candidate_rows`, or `quote_validation_runs` [VERIFIED: bookkeeping_core/database.py].

### Pitfall 4: Using long-tail fallback to smuggle in weakly scoped rows
**What goes wrong:** Candidate coverage numbers improve while false positives rise, which violates the repo’s risk posture [VERIFIED: AGENTS.md, .planning/PROJECT.md].  
**Why it happens:** Context-dependent numeric lines like `50=5.25` tempt planners to default to inferred context even when header scope is ambiguous [VERIFIED: .planning/research/REAL-EXCEPTION-CORPUS.md].  
**How to avoid:** Make fallback deterministic and conservative: require explicit header evidence, preserve source-line lineage, and allow hold/reject/no-publish as the default outcome [VERIFIED: .planning/phases/02.1-real-exception-corpus-candidate-coverage/02.1-CONTEXT.md, bookkeeping_core/quote_validation.py].  
**Warning signs:** A fallback rule emits complete rows without naming the header/span evidence it used [VERIFIED: bookkeeping_core/quote_candidates.py].
