# Phase 02.1: Real Exception Corpus & Candidate Coverage - Research

## Summary

The live exception pool confirms that the next bottleneck is candidate generation, not validator custody or publisher safety. The repository already has most of the mechanical pieces needed to attack this without redesigning the whole stack: a candidate contract from Phase 1, validator custody from Phase 2, a template engine that supports both `strict-section-v1` and `group-parser-v1`, and replay / harvest flows that can save template fragments and immediately re-run candidate generation.

The safest Phase 02.1 cut is:

1. Turn the open exception pool into a refreshable corpus and gold-sample source.
2. Focus implementation on the top-volume chats and top failure shapes first.
3. Improve candidate generation through deterministic section splitting, line classification, scope carry-forward, template bootstrap, and conservative fallback.
4. Package the resulting corpus as replayable fixtures and regression coverage.
5. Keep validator and publisher boundaries unchanged.

## Current Brownfield Findings

### The exception pool already contains the right planning dataset

- The live site exposes `GET /api/quotes/exceptions`, which is already the operational backlog of “real messages the system could not safely digest”.
- The current corpus snapshot shows `226` open exceptions across `25` chats, with the top `8` chats contributing `159` rows.
- Reason split is almost entirely candidate-side:
  - `missing_group_template`: `149`
  - `strict_match_failed`: `77`

Planning consequence: this phase should optimize around the exception pool itself rather than around synthetic parser examples.

### Candidate generation is already a separable layer

- `bookkeeping_core/quote_candidates.py` defines the message-level and row-level candidate contract from Phase 1.
- `bookkeeping_core/quotes.py` already converts parser output into `QuoteCandidateMessage` / `QuoteCandidateRow`.
- Runtime capture persists candidate bundles first, then validation, then exception evidence.
- Replay paths in `bookkeeping_web/app.py` also persist replay candidates first and keep `mutated_active_facts: False`.

Planning consequence: Phase 02.1 does not need to invent a new persistence shape. It needs to improve how more real messages reach the existing candidate contract.

### Template and replay primitives already exist

- `template_engine.py` already supports:
  - `strict-section-v1`
  - `group-parser-v1`
  - result-template preview/save
  - strict-section harvesting
  - multi-section helpers
  - some auto-detection / suggestion utilities
- `bookkeeping_web/app.py` already supports:
  - exception harvest preview/save
  - result preview/save
  - replay with current template
  - group profile update / merge

Planning consequence: the phase should reuse these primitives instead of building a second template system.

## Dominant Real-Sample Failure Shapes

### 1. Missing group template

These cases are usually not ambiguous in the validator sense. They are “good raw boards with no group-specific structure attached yet”.

Implication:
- prioritize template bootstrap and result-template / group-parser skeleton creation for top-volume groups
- reduce manual effort for clean boards that are already structurally rich

### 2. Strict match failed on long mixed boards

These cases usually contain:
- multiple card / country / region sections
- numeric rows that depend on a prior header
- rule lines mixed with quote lines
- `ask` / `先问` / condition lines that should not become facts

Implication:
- candidate generation needs better section splitting and quote-vs-rule classification
- strict matching alone is not enough if header context is not carried into numeric rows

### 3. Mixed quote + rule + manual lines

The corpus repeatedly shows rows like:
- price rows
- usage rules
- “带国家询问后手动报价”
- “不要直接发”
- `ask`

Implication:
- line classification is a first-class requirement
- if not separated early, later phases will keep receiving malformed candidate bundles

### 4. Context-dependent shorthand rows

Examples include:
- `50=5.25`
- `100/150=5.41`
- `+100 xb 5.13`

These rows only make sense relative to a nearby section header or previously established card / country scope.

Implication:
- the parser needs conservative scope carry-forward with explicit evidence
- inferred scope should stay visible so validator and replay can still explain the decision

## Recommended Design For Phase 02.1

### Decision 1: make the corpus refreshable and operational

The corpus should not stay as one static markdown snapshot. The phase should create a durable material loop:

1. pull a targeted slice from the live exception API
2. classify by chat, reason, and content shape
3. curate representative / high-value samples
4. store replayable fixtures plus expected outcomes
5. refresh with new exceptions when coverage is insufficient

This matches the user’s requirement that new real exceptions will keep arriving.

### Decision 2: separate “corpus index” from “gold samples”

Recommended split:

- **Corpus index**
  - broad coverage of the open pool
  - light metadata: chat, reason, pattern bucket, message length/shape
- **Gold samples**
  - curated subset
  - message-level expectation
  - row-level expectations
  - `publishable/held/rejected`
  - `full_snapshot` / `delta_update` / unresolved judgment

This avoids blocking implementation on fully labeling the entire pool.

### Decision 3: improve candidate generation in deterministic layers

Recommended improvement order:

1. **Section splitting**
   - detect board boundaries and section headers
   - preserve section label / defaults
2. **Line classification**
   - `quote`
   - `rule`
   - `inquiry/manual`
   - `header`
   - `noise`
3. **Scope carry-forward**
   - allow naked numeric rows to inherit section scope
   - keep evidence explicit
4. **Template bootstrap**
   - create deterministic group-parser or strict-section skeletons for the highest-volume groups
5. **Conservative fallback**
   - if a line can be partially understood, emit candidate evidence only when scope is justified
   - otherwise leave it in the exception path

### Decision 4: keep model use subordinate

Model use is acceptable only for local exploration or suggestion, for example:
- proposing a section split candidate
- suggesting likely line classification
- helping draft a template fragment

But every accepted improvement must land as:
- explicit code behavior
- explicit fixture expectation
- explicit replayable output

No model-derived policy should become the final rule surface.

## Recommended Artifact Set

Phase 02.1 should likely produce four material classes:

1. **Corpus docs / indexes**
   - planning-facing overview of live pool
2. **Replayable fixture set**
   - raw message samples plus expected candidate/gold outputs
3. **Regression tests**
   - unit and integration tests around template engine, runtime, and replay paths
4. **Code improvements**
   - deterministic candidate-generation hardening for top-volume groups and top failure shapes

## Validation Architecture

Phase 02.1 should be validated at three levels:

1. **Unit validation**
   - template-engine behavior for section splitting, quote-vs-rule detection, shorthand / scope carry-forward, and noise suppression
2. **Integration validation**
   - runtime quote capture persists richer candidate bundles from real-like samples without mutating active facts
   - replay / harvest paths can use the improved template/bootstrap logic and still remain candidate-only
3. **Corpus validation**
   - curated gold samples should show measurable improvement:
     - fewer “whole board collapsed into one exception line” cases
     - more stable candidate bundles
     - no loosening of validator or fact-protection boundaries

The most useful quick suite for this phase is:
- `tests.test_template_engine`
- `tests.test_quote_parser`
- `tests.test_runtime`
- `tests.test_webapp`

Phase-specific additions should land in those same suites unless a dedicated corpus fixture helper is clearly justified.

## Anti-Patterns To Avoid

- Treating the first exception snapshot as the full universe instead of refreshing the corpus when needed
- Chasing coverage by relaxing validator or publisher safety
- Building a second parallel parsing system outside `template_engine.py` / existing quote capture flow
- Using the model as an implicit parser policy engine
- Converting rule lines or `ask` lines into quote rows just because they contain numbers
- Measuring success only by “exceptions decreased” without preserving replayable materials and regression fixtures

## Planning Consequence

Phase 02.1 should be split into a few concrete plans, likely along these boundaries:

1. build the refreshable corpus / gold-sample / fixture baseline
2. harden candidate generation for dominant structural shapes (section split, line classification, scope carry-forward)
3. apply deterministic bootstrap / fallback improvements to the highest-volume groups and wire regression coverage through replay/runtime
