# Phase 02.1: Real Exception Corpus & Candidate Coverage - Context

**Gathered:** 2026-04-14
**Status:** Ready for planning

<domain>
## Phase Boundary

Phase 02.1 exists because the live exception pool proved the current bottleneck is upstream of publisher safety. This phase does two things together: build the real exception corpus and gold-standard material set, and directly improve candidate generation against the highest-volume real customer quote shapes. It does not relax validator custody, does not change publisher ownership, and does not turn model output into business rules.

</domain>

<decisions>
## Implementation Decisions

### Corpus-first execution target
- **D-01:** This phase is not pure research. It must build the real exception corpus and also directly raise candidate hit rate on the highest-volume groups during the same phase.
- **D-02:** Coverage priority is `top 8 exception-heavy chats first`, because they currently represent the majority of the open pool. At the same time, the full open exception pool should receive a lighter-weight index so long-tail shapes are not invisible.
- **D-03:** Live exception pool evidence is the main planning dataset. Planning and implementation should optimize against real collected customer messages, not synthetic examples or prompt-crafted demos.

### Gold-standard answer format
- **D-04:** Gold-standard samples must be message-level and row-level. For each curated sample, the system should capture: valid quote lines, normalized row fields, which rows should be `publishable`, `held`, or `rejected`, and whether the message behaves more like `full_snapshot`, `delta_update`, or remains unresolved.
- **D-05:** A sample is not “gold” if it only marks quote-like lines. It must also express the intended standard fields and the row/message outcomes needed by downstream validator and publisher phases.
- **D-06:** The gold set should be usable for replay, fixtures, and regression tests rather than existing only as prose notes.

### Human authority and candidate suggestions
- **D-07:** For high-frequency groups, the user remains the final authority on gold-standard answers.
- **D-08:** For lower-frequency groups and long-tail samples, the agent may propose candidate answers first, but those answers still require confirmation before they become trusted gold material.
- **D-09:** The project should prefer “agent proposes, human confirms” over waiting for perfect manual labeling of the entire pool before implementation starts.

### Allowed candidate-improvement mechanisms
- **D-10:** This phase may use deterministic templates, deterministic parsing rules, section splitters, and line classifiers.
- **D-11:** Models may be used to explore local candidate strategies, but models may not define business rules, publish rules, validator rules, or final fact outcomes.
- **D-12:** Any model-assisted improvement must still resolve into explicit system logic, persisted evidence, replayable behavior, and regression coverage.

### Success criteria for the phase
- **D-13:** “Done” means both sides move forward together: the material-library loop becomes real, and the top exception-heavy groups show visible candidate-generation improvement against live samples.
- **D-14:** It is not enough to produce a pretty corpus document. The phase must also ship replayable sample sets, fixtures, and concrete code-level candidate improvements.
- **D-15:** It is also not enough to chase hit rate without durable materials. Improvements must be grounded in a reusable corpus and regression surface.

### Long-tail and fallback policy
- **D-16:** New or low-volume groups do not need full coverage in this phase, but they should not stay completely opaque. The phase should add a conservative fallback path that extracts what can be justified and leaves the rest in the exception pool.
- **D-17:** Long-tail fallback may help candidate generation, but if scope is weak or context is missing, the correct outcome remains “do not publish”.
- **D-18:** This phase must not weaken the current “宁可不上墙，不可误上墙” posture just to make long-tail groups look more covered.

### Explicit non-goals
- **D-19:** This phase does not change validator custody boundaries from Phase 2.
- **D-20:** This phase does not centralize publish authority or remove fact-write bypasses; that remains Phase 3 work.
- **D-21:** This phase does not let templates, agents, or prompts directly publish active quote facts.

### the agent's Discretion
- Exact corpus file layout, fixture organization, and whether samples are grouped by chat, shape, or failure mode
- Exact implementation split between section splitter, line classifier, template bootstrap, and deterministic fallback logic
- Exact regression packaging across replay datasets, unit fixtures, and integration tests

</decisions>

<specifics>
## Specific Ideas

- The live exception corpus already shows the dominant shapes: `missing_group_template`, `strict_match_failed`, multi-section boards, quote-plus-rule text, `ask` / manual lines, and header-scoped numeric rows such as `50=5.25`.
- High-value examples include boards that mix several products or regions in one message, plus delta-style updates like `单独更新` where numeric rows depend on a preceding header.
- This phase should treat “real customer quote digestion” as the goal, not “make the parser look more flexible”.
- The system should become better at producing candidate bundles from real exception evidence without giving up explicit evidence, replayability, or conservative validation.

</specifics>

<code_context>
## Existing Code Insights

### Reusable Assets
- `QuoteCandidateMessage` / `QuoteCandidateRow` in `quote_candidates.py`: already define the evidence contract that improved candidate generation must populate
- Runtime and replay candidate persistence in `quotes.py` and `app.py`: already give a durable place to test real-sample candidate improvements
- Validation persistence and `publishable_rows` lookup from Phase 2: already provide a safe downstream surface for measuring whether candidate improvements actually help

### Established Patterns
- The system already preserves raw message evidence and replay lineage, so corpus work should reuse those boundaries instead of inventing a parallel labeling path
- Candidate, validation, and fact layers are intentionally separate. This phase should improve the first layer while preserving the later ones

### Integration Points
- Exception-pool API and stored `quote_parse_exceptions`
- Group profile / template workflow in the quote web UI
- Runtime parse flow and replay flow that already materialize candidate bundles
- Unit and integration test suites in `wxbot/bookkeeping-platform/tests/`

### Risk Hotspots
- High-frequency groups still fail before validator can help, so section splitting and context carry-forward need to improve without accidentally promoting rule text into candidate rows
- Long board posts often mix quote rows, operational rules, `ask` rows, and multiple region/product sections; naive regex expansion will likely raise false positives
- There is a temptation to use the model as an implicit parser policy engine; this phase must resist that and convert any useful strategy into explicit, testable system behavior

</code_context>
