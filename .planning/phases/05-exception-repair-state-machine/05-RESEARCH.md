## User Constraints (from CONTEXT.md)

### Locked Decisions
- **D-01:** Every failed, partial, rejected, or unclassifiable quote parse must become a durable repair case, not just a disposable exception row.
- **D-02:** The repair case is the canonical object for later subagent remediation, replay comparison, and escalation. Later phases should not need to reconstruct context ad hoc from scattered tables.
- **D-03:** Phase 05 should build on the existing `quote_parse_exceptions` evidence instead of inventing a parallel exception-capture pipeline disconnected from runtime and replay.

### System-led failure packaging
- **D-04:** Repair cases must be generated by system-detected failures such as `missing_group_template`, `strict_match_failed`, validator `held`/`rejected`, zero-publish candidate runs, or unresolved message outcomes.
- **D-05:** The system, not the model, decides when a repair case is opened, updated, retried, escalated, or closed.
- **D-06:** Models and subagents may later inspect repair cases, but they must consume a standard package assembled by the system rather than freeform logs.

### Replay-critical evidence must travel with the repair case
- **D-07:** A repair case must carry enough context to rerun and compare a message without manually rediscovering inputs.
- **D-08:** Minimum repair evidence includes: raw message, source line/unmatched evidence, current group profile/template version, candidate bundle reference, validator run reference, replay baseline, failure reason(s), and prior attempt history.
- **D-09:** The repair package must preserve owning group identity and group-profile versioning because this project evolves one group grammar at a time rather than chasing a global parser.

### Before/after replay is the proof surface
- **D-10:** Phase 05 must define a deterministic baseline replay for the owning group profile so a later proposed fix can be judged as “better”, “same”, or “worse”.
- **D-11:** Before/after replay comparison is required because fixing one exception is not enough; the system must know whether the group grammar improved safely.
- **D-12:** The comparison surface should stay candidate/validator oriented in this phase. It must not imply that a better replay automatically updates active facts.

### Cumulative history and bounded escalation
- **D-13:** Repair work is iterative. The system must retain cumulative attempt history and failure-log stacking instead of overwriting earlier failures.
- **D-14:** State should be explicit and machine-readable, supporting at minimum: newly detected, packaged, ready for attempt, attempt failed, attempt succeeded, escalated, and closed/resolved.
- **D-15:** Phase 05 should make repeated failure visible and escalation-ready, but the actual bounded subagent loop belongs to Phase 06.

### Group-first remediation philosophy
- **D-16:** The system should preserve owning-group context so later remediation can prefer group profile / group section / bootstrap fixes before shared parser changes.
- **D-17:** A repair case is not just “an error”. It is evidence that a specific group's grammar is incomplete or a shared deterministic rule is insufficient.
- **D-18:** Nothing in this phase should push the architecture back toward a global万能解析器 or prompt-defined rules.

### Explicit non-goals
- **D-19:** Phase 05 does not create the auto-remediation loop itself; it prepares the state machine substrate for Phase 06.
- **D-20:** Phase 05 does not change guarded publisher ownership; active quote mutation remains out of scope until Phase 03.
- **D-21:** Phase 05 does not redefine snapshot semantics or operator confirmation flows; those remain later-phase concerns.

### Claude's Discretion
- Whether repair-case state is implemented as additive columns on `quote_parse_exceptions`, a dedicated `quote_repair_cases` table, or a hybrid with linked attempt tables
- Exact replay-baseline representation as long as it is durable, deterministic, and tied to owning group profile/template lineage
- Exact API/test surface for before/after comparison, provided it is consumable by later remediation and operator flows

### Deferred Ideas (OUT OF SCOPE)
- Bounded subagent remediation attempts, retry cap, and failure-log-driven repair proposals remain Phase 06
- Guarded publisher and active-fact protection remain Phase 03
- Snapshot/delta execution semantics remain Phase 04
- Operator-facing repair workbench remains later UI/operator phases, though Phase 05 should create the data model they will consume

## Summary

The brownfield already has the right raw ingredients for Phase 05: `quote_documents` persist message-level candidate bundles, `quote_candidate_rows` persist row evidence, `quote_validation_runs` and `quote_validation_row_results` persist validator-owned outcomes, `quote_parse_exceptions` persists failure rows, and `quote_group_profiles` persists the current owning group grammar. [VERIFIED: codebase `wxbot/bookkeeping-platform/sql/postgres_schema.sql`; `wxbot/bookkeeping-platform/bookkeeping_core/database.py`] What it does not have is a durable object that binds those ingredients into one repair-case lifecycle with immutable baseline evidence, machine-readable state, and cumulative attempt history. [VERIFIED: codebase `wxbot/bookkeeping-platform/bookkeeping_core/database.py`; `.planning/phases/05-exception-repair-state-machine/05-CONTEXT.md`]

`quote_parse_exceptions` is currently a log-and-resolution table, not a state machine. [VERIFIED: codebase `wxbot/bookkeeping-platform/sql/postgres_schema.sql`; `wxbot/bookkeeping-platform/bookkeeping_core/database.py`] It stores `resolution_status`, `resolution_note`, and `resolved_at`, but it has no repair-case identifier, no link to a validator run, no immutable group-profile snapshot, no replay-baseline record, and no attempt stack. [VERIFIED: codebase `wxbot/bookkeeping-platform/sql/postgres_schema.sql`; `wxbot/bookkeeping-platform/bookkeeping_core/database.py`] That makes additive columns on `quote_parse_exceptions` too cramped for Phase 05, while a pure parallel table with no reference back to exception rows would violate the project decision to build on the existing exception evidence. [VERIFIED: codebase `.planning/phases/05-exception-repair-state-machine/05-CONTEXT.md`; `wxbot/bookkeeping-platform/bookkeeping_core/database.py`]

The recommended brownfield direction is a hybrid: keep `quote_parse_exceptions` as the raw failure-event surface and add a dedicated `quote_repair_cases` table plus a linked `quote_repair_case_attempts` table. [VERIFIED: codebase `.planning/phases/05-exception-repair-state-machine/05-CONTEXT.md`; `wxbot/bookkeeping-platform/bookkeeping_core/database.py`; `wxbot/bookkeeping-platform/bookkeeping_web/app.py`] The case row should own lifecycle state and immutable packaging snapshots; the attempt rows should own baseline replay and later remediation replay history; existing `quote_documents`, `quote_validation_runs`, and `quote_validation_row_results` should remain the canonical candidate/validator truth instead of being duplicated into another mutable JSON blob. [VERIFIED: codebase `wxbot/bookkeeping-platform/bookkeeping_core/database.py`; `wxbot/bookkeeping-platform/tests/test_webapp.py`]

**Primary recommendation:** Implement Phase 05 as `quote_parse_exceptions` + `quote_repair_cases` + `quote_repair_case_attempts`, with one repair case per originating exception row and one immutable baseline replay attempt per case before any later remediation. [VERIFIED: codebase `.planning/ROADMAP.md`; `.planning/REQUIREMENTS.md`; `wxbot/bookkeeping-platform/bookkeeping_web/app.py`]

### Pattern 1: Failure Event -> Repair Case -> Attempt History
**What:** Keep `quote_parse_exceptions` as the event row, then open exactly one `quote_repair_cases` row per originating exception, and store baseline plus future retries in `quote_repair_case_attempts`. [VERIFIED: codebase `wxbot/bookkeeping-platform/bookkeeping_core/database.py`; `.planning/phases/05-exception-repair-state-machine/05-CONTEXT.md`]
**When to use:** For all Phase 05 exception types, including `missing_group_template`, `strict_match_failed`, validator-held/rejected outcomes, and zero-publish message outcomes that already produce candidate/validation evidence. [VERIFIED: codebase `.planning/phases/05-exception-repair-state-machine/05-CONTEXT.md`; `.planning/REQUIREMENTS.md`; `wxbot/bookkeeping-platform/bookkeeping_core/quotes.py`; `wxbot/bookkeeping-platform/bookkeeping_core/quote_validation.py`]
**Example:**
```python
# Source pattern: existing brownfield flow records candidate bundle, then validator run,
# then exception rows, without mutating active quote facts.
document_id = self.db.record_quote_candidate_bundle(candidate=candidate)
validation_run = validate_quote_candidate_document(
    quote_document_id=document_id,
    run_kind=candidate.run_kind,
    candidate_document=candidate,
    candidate_rows=self.db.list_quote_candidate_rows(quote_document_id=document_id),
)
validation_run_id = self.db.record_quote_validation_run(validation_run=validation_run)
exception_id = self.db.record_quote_exception_unless_suppressed(...)
```
Source: [VERIFIED: codebase `wxbot/bookkeeping-platform/bookkeeping_core/quotes.py`]

### Pattern 3: Baseline Replay As A Candidate-Only Attempt
**What:** Represent the "before" baseline as its own persisted replay document + validation run pair, not as an ephemeral diff in an API response. [VERIFIED: codebase `wxbot/bookkeeping-platform/bookkeeping_web/app.py`; `wxbot/bookkeeping-platform/tests/test_webapp.py`] 
**When to use:** Once a repair case is packaged and the current group profile/template is known. [VERIFIED: codebase `.planning/phases/05-exception-repair-state-machine/05-CONTEXT.md`; `wxbot/bookkeeping-platform/bookkeeping_web/app.py`]
**Example:**
```python
replay_document_id = db.record_quote_candidate_bundle(candidate=candidate)
validation_run = validate_quote_candidate_document(
    quote_document_id=replay_document_id,
    run_kind="replay",
    candidate_document=candidate,
    candidate_rows=db.list_quote_candidate_rows(quote_document_id=replay_document_id),
)
validation_run_id = db.record_quote_validation_run(validation_run=validation_run)
```
Source: [VERIFIED: codebase `wxbot/bookkeeping-platform/bookkeeping_web/app.py`]

### Anti-Patterns to Avoid
- **Case state on `quote_parse_exceptions` alone:** The current exception table only models `resolution_status` and note text; extending it into a full attempt stack will mix event logs with workflow state and make one-case-per-exception enforcement awkward. [VERIFIED: codebase `wxbot/bookkeeping-platform/sql/postgres_schema.sql`; `wxbot/bookkeeping-platform/bookkeeping_core/database.py`]
- **Repair packages that duplicate candidate/validator rows wholesale:** `quote_candidate_rows` and `quote_validation_row_results` are already durable truth. Phase 05 should store foreign keys plus frozen snapshots only where source rows are mutable or API-only, such as group-profile config and replay remaining lines. [VERIFIED: codebase `wxbot/bookkeeping-platform/bookkeeping_core/database.py`; `wxbot/bookkeeping-platform/bookkeeping_web/app.py`]
- **Any repair attempt that touches `quote_price_rows`:** current runtime and replay tests prove candidate/replay flows can remain fact-neutral; Phase 05 must preserve that boundary. [VERIFIED: codebase `wxbot/bookkeeping-platform/tests/test_runtime.py`; `wxbot/bookkeeping-platform/tests/test_webapp.py`; `.planning/ROADMAP.md`]

### Lifecycle packaging rule
At repair-case creation time, package the current `quote_parse_exceptions` row, the originating `quote_documents` row, the latest validator run for that document, the current `quote_group_profiles` row if one exists, and a frozen JSON snapshot of the mutable grammar fields. [VERIFIED: codebase `wxbot/bookkeeping-platform/bookkeeping_core/database.py`; `wxbot/bookkeeping-platform/bookkeeping_web/app.py`] Do not copy candidate rows or validator row results into the case table; reference them by ID and query them on demand. [VERIFIED: codebase `wxbot/bookkeeping-platform/bookkeeping_core/database.py`] Copy only values that are otherwise mutable or currently API-only: raw message text, unresolved source-line slice, profile snapshot JSON, and replay `remaining_lines`. [VERIFIED: codebase `wxbot/bookkeeping-platform/bookkeeping_web/app.py`; `wxbot/bookkeeping-platform/bookkeeping_core/database.py`]

## Replay Baseline Representation

The baseline should not be a note string or a diff blob. [VERIFIED: codebase `.planning/phases/05-exception-repair-state-machine/05-CONTEXT.md`] The most durable brownfield shape is:

| Baseline element | Representation | Why |
|------------------|---------------|-----|
| Original failing evidence | `origin_exception_id`, `origin_quote_document_id`, `origin_validation_run_id` on `quote_repair_cases` | Preserves what happened when the failure entered the pool. [VERIFIED: codebase `wxbot/bookkeeping-platform/bookkeeping_core/quotes.py`; `wxbot/bookkeeping-platform/bookkeeping_core/database.py`] |
| Immutable "before" proof | Attempt row `attempt_kind='baseline'` with its own replay `quote_document_id` and `validation_run_id` | Gives a deterministic candidate/validator baseline under the packaged grammar state, not just the original runtime conditions. [VERIFIED: codebase `wxbot/bookkeeping-platform/bookkeeping_web/app.py`; `.planning/phases/05-exception-repair-state-machine/05-CONTEXT.md`] |
| Mutable grammar capture | `profile_snapshot_json` on the baseline attempt | Needed because `quote_group_profiles` has no built-in version history. [VERIFIED: codebase `wxbot/bookkeeping-platform/bookkeeping_core/database.py`; `wxbot/bookkeeping-platform/sql/postgres_schema.sql`] |
| Unmatched residue | `remaining_lines_json` on the baseline attempt | `_replay_latest_quote_document_with_current_template(...)` currently returns `remaining_lines` in-memory only. That signal must be persisted for later comparison. [VERIFIED: codebase `wxbot/bookkeeping-platform/bookkeeping_web/app.py`] |
| Comparison surface | `attempt_summary_json` with row counts, message decision, exception count, and strict candidate/validator deltas vs baseline | Lets later remediation decide better/same/worse without touching active quote facts. [VERIFIED: codebase `.planning/ROADMAP.md`; `wxbot/bookkeeping-platform/tests/test_webapp.py`] |

### Recommended baseline workflow
1. Open repair case from the exception row. [VERIFIED: codebase `.planning/REQUIREMENTS.md`; `wxbot/bookkeeping-platform/bookkeeping_core/database.py`]
2. Freeze current group-profile fields into case snapshot JSON. [VERIFIED: codebase `wxbot/bookkeeping-platform/bookkeeping_core/database.py`]
3. Run one baseline replay through the existing replay helper or an extracted shared helper that preserves the current candidate-only behavior. [VERIFIED: codebase `wxbot/bookkeeping-platform/bookkeeping_web/app.py`; `.planning/phases/01-candidate-contract-foundation/01-03-PLAN.md`] 
4. Persist that replay as attempt `0` with `attempt_kind='baseline'`, its replay `quote_document_id`, `validation_run_id`, `remaining_lines_json`, and `mutated_active_facts` implied false because the helper never writes `quote_price_rows`. [VERIFIED: codebase `wxbot/bookkeeping-platform/tests/test_webapp.py`; `wxbot/bookkeeping-platform/tests/test_runtime.py`] 
5. Later remediation attempts compare their replay document/run against the baseline attempt, not against live board state. [VERIFIED: codebase `.planning/phases/05-exception-repair-state-machine/05-CONTEXT.md`; `.planning/ROADMAP.md`]

## Common Pitfalls

### Pitfall 1: Treating `quote_parse_exceptions` as the repair-case table
**What goes wrong:** The exception row ends up carrying workflow state, baseline evidence, retry notes, and suppression metadata at the same time. [VERIFIED: codebase `wxbot/bookkeeping-platform/sql/postgres_schema.sql`; `wxbot/bookkeeping-platform/bookkeeping_core/database.py`] 
**Why it happens:** The current UI already reads `quote_parse_exceptions`, so it is tempting to keep stuffing fields into it. [VERIFIED: codebase `wxbot/bookkeeping-platform/bookkeeping_web/app.py`] 
**How to avoid:** Keep exception rows as seed events and put lifecycle/attempt state in dedicated repair tables. [VERIFIED: codebase `.planning/phases/05-exception-repair-state-machine/05-CONTEXT.md`] 
**Warning signs:** One row needs arrays of attempts, multiple baseline IDs, or state that conflicts with `resolution_status`. [VERIFIED: codebase `wxbot/bookkeeping-platform/sql/postgres_schema.sql`] 

### Pitfall 2: Storing only live `quote_group_profiles` links with no frozen snapshot
**What goes wrong:** A later profile edit changes the meaning of an old repair case, so before/after proof is no longer reproducible. [VERIFIED: codebase `wxbot/bookkeeping-platform/bookkeeping_core/database.py`] 
**Why it happens:** `upsert_quote_group_profile(...)` updates the same row in place and there is no version table today. [VERIFIED: codebase `wxbot/bookkeeping-platform/bookkeeping_core/database.py`; `wxbot/bookkeeping-platform/sql/postgres_schema.sql`] 
**How to avoid:** Persist profile snapshot JSON on case open and on every attempt. [VERIFIED: codebase `.planning/phases/05-exception-repair-state-machine/05-CONTEXT.md`] 
**Warning signs:** Old cases display a newer `template_config` than the one used when the case was opened. [VERIFIED: codebase `wxbot/bookkeeping-platform/bookkeeping_core/database.py`] 

### Pitfall 3: Using baseline comparison that depends on active board state
**What goes wrong:** Repair proof becomes entangled with publisher behavior and later snapshot semantics, which belong to Phases 03 and 04. [VERIFIED: codebase `.planning/ROADMAP.md`; `.planning/phases/05-exception-repair-state-machine/05-CONTEXT.md`] 
**Why it happens:** The repo already has board and exception pages, so it is easy to think "better wall output" is the baseline. [VERIFIED: codebase `wxbot/bookkeeping-platform/bookkeeping_web/app.py`] 
**How to avoid:** Compare only candidate rows, validation decisions, remaining lines, and replay exception counts in Phase 05. [VERIFIED: codebase `.planning/phases/05-exception-repair-state-machine/05-CONTEXT.md`; `wxbot/bookkeeping-platform/tests/test_webapp.py`] 
**Warning signs:** A Phase 05 API tries to read or mutate `quote_price_rows` to decide whether a repair succeeded. [VERIFIED: codebase `wxbot/bookkeeping-platform/tests/test_runtime.py`; `wxbot/bookkeeping-platform/tests/test_webapp.py`] 

### Pitfall 4: Letting Phase 05 absorb fixes into group profiles automatically
**What goes wrong:** Phase boundaries blur and the repo gets an implicit remediation loop before bounded attempt governance exists. [VERIFIED: codebase `.planning/ROADMAP.md`; `.planning/REQUIREMENTS.md`] 
**Why it happens:** Current web handlers such as harvest-save and result-save already update `quote_group_profiles` directly. [VERIFIED: codebase `wxbot/bookkeeping-platform/bookkeeping_web/app.py`] 
**How to avoid:** Phase 05 should package state and record attempts; actual auto-remediation and promotion order stay in Phase 06. [VERIFIED: codebase `.planning/phases/05-exception-repair-state-machine/05-CONTEXT.md`; `.planning/ROADMAP.md`] 
**Warning signs:** A successful repair-case attempt mutates `template_config` or `parser_template` without an explicit later-phase absorption flow. [VERIFIED: codebase `wxbot/bookkeeping-platform/bookkeeping_web/app.py`]
